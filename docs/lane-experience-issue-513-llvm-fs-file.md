# Lane experience — issue #513 (LLVM emit missing fs/file runtime symbols)

## Scope as planned

Close the LLVM-backend link error for programs using `fs/file` byte ops:

```
… error: use of undefined value '@kai_file_read_bytes'
```

The C backend has always declared `kai_prelude_file_read_bytes` /
`kai_prelude_file_write_bytes` via `stage0/runtime.h`. The LLVM emit
path (`emit_program_llvm` in `stage2/compiler.kai`) did not list them
in its prelude table, so calls to `file_read_bytes(p)` fell through
to the user-symbol minter (`@kai_file_read_bytes`) — which has no
definition — and `clang` rejected the IR at link time.

Workaround in place since #512 / commit ebf1239: `scripts/build-release.sh`
builds `tools/kai-pkg` with `KAI_BACKEND=c` explicitly. The lane brief
expected this workaround to fall away with the fix.

## Scope as shipped

Three independent commits, all on this branch:

1. **`fix(emit): declare fs/file runtime symbols in LLVM IR prologue (closes #513)`**
   - Add `LP("file_read_bytes", 1, None)` + `LP("file_write_bytes", 2, None)`
     to `llvm_prelude_table` (stage2/compiler.kai ~line 40903).
   - Add `declare %KaiValue* @kaix_prelude_file_read_bytes(%KaiValue*)`
     and `declare %KaiValue* @kaix_prelude_file_write_bytes(%KaiValue*, %KaiValue*)`
     to the IR prologue (~line 40509).
   - Add `kaix_prelude_file_read_bytes` / `kaix_prelude_file_write_bytes`
     wrappers in `stage0/runtime_llvm.c` (mirror the existing
     `kaix_prelude_file_exists` / `_delete` / `_rename` triple at line 290).

2. **`test(fs/file): cross-backend parity fixture`**
   - New fixture `examples/stdlib/fs_file_bytes_llvm.kai` writes 4 bytes
     (including 0x00 and 0xFF) then reads them back and prints them.
     Stays at the flat `file_read_bytes` / `file_write_bytes` surface
     intentionally — see the qualified-call collision filed as a
     separate follow-up issue (#524).
   - Golden: `examples/stdlib/fs_file_bytes_llvm.out.expected`.
   - Wired into `tools/test-llvm-driver.sh` `$FIXTURES` so the parity
     gate runs whenever the LLVM driver test runs (every tier1 sweep).

## Scope reconsidered: workaround NOT removed

The brief expected Part D (drop `KAI_BACKEND=c` in `scripts/build-release.sh`)
to fall away naturally. **It didn't.** Investigation:

1. After the declaration fix, `bin/kai build --backend=llvm
   tools/kai-pkg/main.kai -o tools/kai-pkg/kai-pkg` now succeeds (no
   more undefined symbol). The link error #513 was about is closed.
2. But the resulting `kai-pkg` binary **segfaults at runtime** on
   subcommands that read deps (`paths`, `install`, …). Confirmed by
   running `bash scripts/build-release.sh` end-to-end: the smoke
   test's manifest-mode project triggers `kai-pkg paths` via the
   `bin/kai` wrapper, which segfaults under LLVM and falls through.
3. Same `kai-pkg` binary built with `KAI_BACKEND=c` runs cleanly.

The segfault is a **separate, pre-existing runtime gap** that #513
did not promise to fix and that the original workaround was actually
hiding (the link error was the visible part — kai-pkg never linked,
so its runtime behaviour was never exercised). Removing the
`KAI_BACKEND=c` prefix now would ship a release tarball whose
`kai-pkg` segfaults on every `kai run <project-with-kai.toml>` call.

Decision: **keep the workaround**, rewrite the comment block to
distinguish the two failure modes, and file the runtime crash as a
follow-up to #513. Same for the manifest-mode smoke test on line 167.

## Symbol diff: C-path prelude vs LLVM-path prelude

`LP(...)` in `llvm_prelude_table`. Generated by:

awk '/^    LP\(/ { ... }' stage2/compiler.kai | sort > /tmp/lp.txt

29 LLVM-side gaps in addition to the two #513 closed. Every one is
fixable in three one-liners (LP entry + IR `declare` + `kaix_*`

in `runtime.h`. The lane brief explicitly said NOT to bundle Part E
fixes unless trivial — these are mechanically trivial individually
but add up to ~100 lines across three files for math+mailbox. Two
focused follow-ups (#522, #523) preserve scope discipline.

## Follow-ups filed

| Issue | Title | Scope |
|-------|-------|-------|
| #522  | `stdlib/math: real_* libm bindings missing from LLVM emit prelude table` | 21 ops |
| #523  | `stdlib mailbox runtime missing from LLVM emit prelude table` | 7 ops |
| #524  | `module-qualified file.read_bytes shadows prelude read_bytes(n) under LLVM` | 1 line, but structural |

#524 was a surprise: it surfaces only because `fs/file.kai`'s wrapper
exports a `read_bytes(path)` name that collides with the prelude's
unqualified `read_bytes(n)` (Stdin). The LLVM emitter's `EModCall`
arm short-circuits to `@kaix_prelude_read_bytes` whenever
`llvm_is_prelude_fn(fname)` is true, ignoring the module qualifier
that the resolver had already validated. The C backend's `EModCall`
arm uses `c_sym(fname, Some(mod))` first, so it correctly emits
`kai_file__read_bytes`. The fix is two characters (delete the
prelude shortcut from the `EModCall` arm), but it changes semantics
for any program that depends on the current shadowing behaviour, so
it needs its own lane.

## Structural surprises the brief did not anticipate

1. **`scripts/build-release.sh` was hiding two bugs, not one.** The
   link error was the visible blocker; the runtime segfault was
   masked entirely because kai-pkg had never linked. Removing the
   workaround naively (as the brief drafted Part D) would have
   shipped a broken release.

2. **The qualified-call collision (#524) is a different shape than
   the missing-declaration gap.** Both manifest as "LLVM does X but
   C does Y" for fs/file, so a glance suggests one root cause. They
   are not — one is a missing entry in the prelude table, the other
   is an over-eager shortcut in the qualified-call emitter. Bundling
   them would have produced a lane that touched two unrelated bug
   shapes.

3. **The lane brief's Part E ("audit other stdlib modules") found
   real gaps, not theoretical ones.** 29 symbols total across math
   + mailbox. The brief was right to ask, wrong to suggest most of
   them would be trivially one-line — individually yes, but cumulative
   they exceed the scope of a "close #513" PR.

## Fixtures added

  for `file_write_bytes` / `file_read_bytes` at the flat surface

## Coverage gaps

The qualified surface (`file.read_bytes` / `file.write_bytes` via the
`fs.file` wrapper) is NOT covered by the fixture because #524 still
breaks it. The fixture comment block explains the choice. When #524
closes, the fixture should be updated (or a sibling fixture added) to
exercise the qualified path.

## Cost vs estimate

Brief estimate: 1–2 days.
Real time-on-task: ~1.5 hours.

The actual code change was ~10 lines. The remaining time went to:
- Reproducing the bug (5 min).
- Auditing the full EP-vs-LP diff (15 min).
- Discovering and triaging the kai-pkg runtime segfault (~30 min,
  including the dev-mode investigation that ruled out a regression).
- Discovering the #524 qualified-call collision while validating
  paritied output for the fixture (~15 min).
- Writing fixtures + retros + the three follow-up issues (~30 min).

The brief underestimated the audit + triage component. The fix
itself was indeed mechanical, but a #513-style lane on a complex
emitter surface routinely surfaces 2–3 adjacent bugs that need
either inline fixes or proper triage. Treating "audit other stdlib
modules" as an optional Part E in the brief invited it to be
skimmed; the right framing for next time is "audit is a first-class
deliverable, expect to find 0–3 follow-ups, plan ~30 min to file
them properly."

## What would make the next "close a runtime gap" lane easier

A static `audit-runtime-coverage.sh` tool that reads `prelude_table`
+ `llvm_prelude_table` from `stage2/compiler.kai` directly and
diffs the two. The awk one-liners above are reusable; making them
a checked-in tool would surface gaps as part of every PR that
edits either table.
