# RB-tree vs Koka/C/C++ — 2026-06-02 (post-reuse-cascade, Koka column landed)

First **apples-to-apples Koka measurement** of the canonical Perceus
red-black tree workload (`examples/perceus/rb_tree_bench.kai`, 1 M random
inserts), on the same machine, with the same keystream. This closes the
open *next-step #2* from the 2026-05-28 pre-Koka baseline ("install Koka,
run the same §7 workload") — which was named but never run with numbers.
(That 2026-05-28 doc, and the earlier 05-08/05-09 rb-tree measurements,
were deleted once superseded; their figures are quoted inline here.)

It also retires the 2026-05-28 headline. That doc reported **3637 ms /
294 MiB / 14.78× C** as a *pre-cascade baseline*. The reuse-in-place
cascade that followed (`c44a445` → `e345393`, eight perf commits) moved
the wall ~5.4× and the RSS ~5.3×. The 14.78× number is dead; do not cite
it as current.

This is a **measurement doc**: no compiler or stdlib code was changed in
this lane. HEAD is `fcffd3c`.

| Variable | Value |

| C++ reference | `clang++ -std=c++17 -O2` (`std::set<int64_t>`) |

The Koka source is `koka/test/bench/koka/rbtree-ck.kk` (the Lean4/Koka
benchmark that `examples/perceus/rb_tree.kai` was itself adapted from)
with the algorithm — `ins` / `balance1` / `balance2` / `insert` — taken
verbatim, and **only the driver replaced** so the keystream matches
kaikai/C exactly (same LCG, `value:bool → value:int`, `make-tree →
fill-loop`). All four implementations produce `size: 1000000` (every key
in the keystream is unique), so the measured work is identical.

## The cascade that retired the 14.78× number

The trajectory, each step cited from the commit that shipped it:

| Commit | Lever | Effect (as the commit measured it) |
| --- | --- | --- |
| `c44a445` | tagged-Int + reuse-token + borrow | 14.78× C → ~5× C wall |
| `5ca630d` | arm-binder move + linear reuse-token | 6.2× C → 4.4× C wall |
| `141b1b7` | consume-map + tl-gated cons-overwrite | RSS **38.6× C → 2.29× C** (1920 → 114 MB trace live-peak), reuse_in_place 5.6M → 19.6M |
| `e345393` | borrow-pure unique branch (Koka move model) | incref/decref −43%, wall 0.63s → **0.54s** |

`free_total == alloc_total` (spine fully reclaimed) holds at HEAD —
Perceus garbage-free is real on this workload now, which is why the
process RSS sits at **1.17× C** instead of the 6.26× C of the 05-28 doc.

## Koka is the yardstick, and the gap is now a constant

With the same algorithm and keystream, Koka runs at **1.12× C** and
kaikai at **2.64× C** — i.e. **~2.36× Koka**. This is no longer the ~14×
chasm of the 05-28 baseline; it is a constant-factor gap, and the
instruction counts say exactly where it lives:

- **kaikai retires 8.37 G instructions vs Koka's 1.09 G — 7.7×**, for a
  bit-identical tree. Allocs are already at parity (the reuse retro:
  `allocs unchanged 6.30M`, an RC-traffic lever now exhausted) and
  reuse-in-place is firing (`5.6M → 19.6M`). The 7.7× instruction gap is
  **not** allocation and **not** missing reuse.
- It is the boxed↔unboxed `Int` frontier. Every key comparison and every
  argument pass still routes a small `Int` through a `kai_int` box; Koka
  represents a small `Int` as a value-immediate (tagged pointer, never a
  heap box, register-move field read). This is the single remaining
  large lever, and it is the one the 05-28 doc already named:
  **value-immediate `Int`**.

## Reproduction

```sh
# kaikai / C / C++ table (existing wrapper):
bash examples/perceus/rb_tree_bench.sh 7

# Koka column (algorithm verbatim from rbtree-ck.kk, LCG driver):
#   driver source: /tmp/rbcmp/rbbench.kk in the measuring session;
#   to regenerate, take koka/test/bench/koka/rbtree-ck.kk, change
#   value:bool -> value:int, and replace make-tree/main with the
#   fill-loop(LCG) driver from rb_tree_bench.kai.
koka -O2 -o /tmp/rb_kk rbbench.kk
for n in rb_kai rb_c rb_cxx rb_kk; do /usr/bin/time -l ./$n; done   # real + maximum resident
```

## What is settled vs open

- **Settled:** wall 2.64× C / 2.36× Koka, RSS 1.17× C, garbage-free. The
  reuse cascade is done; further reuse work is not the lever (the retro's
  `reuse_freed` note: "Do not pursue" — C and Koka pay the same random-
  descent cache miss).
- **Open:** value-immediate `Int`. The 7.7× instruction gap to Koka is
  boxing traffic at the `RBNode`-with-pointer-children frontier, where
  Phase 3 signature-unboxing does not reach. Closing it is what moves
  2.64× C toward Koka's 1.12× C.
