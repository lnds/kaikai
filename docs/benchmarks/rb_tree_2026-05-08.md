# Red-black tree insert benchmark — re-measurement on 0.44.1

Re-run of the canonical Perceus red-black tree workload (Reinking,
Xie, de Moura, Leijen, POPL'21 §7) against the current `main` after
the night-of-2026-05-07 RC discipline + resolver fixes landed
(through PR #342, perceus exit-drops for SLet pass-through).

The original measurement is PR #305 (closed without merge). Its
fixture, driver, C reference, and wrapper script are reused
verbatim — the only thing that changed is the kaikai compiler
behind it.

This document records the **second** measurement, ~24 h after the
first, on the same host.

## Pins

- HEAD: `3c0d2bff890ae4df48d6d2bfbf61ff24299c584f` (post-#342)
- Branch: `rb-tree-remeasure` rebased onto `origin/main`
- VERSION: `0.44.1`
- Host: Apple M4 Pro, macOS 26.4.1, Apple clang 21.0.0
- Build flags: `clang -std=c99 -O2` (kaikai default in `bin/kai`),
  `clang -std=c11 -O2` for the C reference,
  `clang++ -std=c++17 -O2` for the C++ reference.
- Stages rebuilt clean: `make -C stage{0,1,2} clean && make tier0`
  (selfhost byte-identical).
- Method: 7 runs each, `/usr/bin/time -l` for wall + RSS, median.

(The PR #305 doc text reads "Apple M1, macOS 25.4.0, clang 17". The
underlying machine is the same Apple M4 Pro / macOS 26.4.1 host —
that line in #305 was a hand-typed error and is corrected here.
RSS counts and reuse traces, which are hardware-independent, match
#305 to within rounding, consistent with "same host, same workload,
different kaikai".)

## Workload (unchanged from #305)

- Pure-functional Okasaki red-black tree over `Int → Int`
  (`examples/perceus/rb_tree.kai`).
- 1,000,000 random inserts driven by a Numerical-Recipes LCG
  (multiplier 1664525, increment 1013904223, modulus 2^31 - 1)
  seeded with 1.
- Driver: `examples/perceus/rb_tree_bench.kai`.
- C reference (intrusive RB with parent pointers) +
  C++ `std::set<int64_t>`: `examples/perceus/rb_tree_bench_c.c`.
- Same keystream feeds every implementation — apples-to-apples.

Wall-clock dropped on all three implementations, but C and C++
fell faster (−28 / −26 %) than kaikai (−12 %). The most likely
explanation is between-session noise on the host (thermal,
scheduler, concurrent processes) — the absolute kaikai number

sample, so the pre/post gap is sensitive to which run got
captured as the median. The **counters** (next section) move by
< 5 × 10⁻⁷ across the whole window, so no structural regression
is plausible.

The kaikai/reference ratios drifted up (+23 % vs C, +19 % vs C++).
We treat this drift as "same magnitude, different noise floor"
rather than a kaikai regression: every PR in the merge window
that touches benchmark-relevant code paths (#324 closure capture,
#333 TCO, #342 SLet exit-drops) reduces work, not adds it; and
the RC counters confirm the workload's allocation profile is
unchanged.

## Pre #342 → Post #342 (within this same lane, 2026-05-08)

This lane originally measured `63248b2` (post-#339, pre-#342) at
the start of the day, then rebased onto `3c0d2bf` (post-#342)
and re-measured. Both passes ran on the same host, same binaries
for C / C++:

| Metric                | Pre #342 (63248b2)    | Post #342 (3c0d2bf)   | Δ abs     | Δ %       |
| --------------------- | --------------------: | --------------------: | --------: | --------: |
| kaikai median wall    |             2 305 ms  |             2 642 ms  | +337 ms   | +14.6 %   |
| kaikai median RSS     |             1 929 MB  |             1 929 MB  | ±0 MB     |  0.0 %    |
| C median wall         |               232 ms  |               222 ms  | −10 ms    | −4.3 %    |
| C++ median wall       |               231 ms  |               216 ms  | −15 ms    | −6.5 %    |
| kaikai variance spread|           583 ms (25 %)|           101 ms (4 %)|           | tighter   |

The kaikai wall went **up** 14.6 %. Two facts make this almost
certainly noise rather than a #342 regression:

1. **The pre-#342 sample had a 25 % spread** (2247 → 2830 ms).
   Two of seven runs were tail-heavy (2727, 2830). The median
   landed at 2305 ms. The post-#342 sample has a 4 % spread,
   tightly clustered around 2625 ms. If we drop the two
   tail-heavy outliers from the pre-#342 sample, its mean is
   2287 ms — still well below 2625 ms — so the move isn't
   explained by outlier alignment alone.
2. **The RC counters (below) moved by 8 cells out of 28.6 M**
   (2.8 × 10⁻⁷), and `live_peak` is byte-identical. Whatever
   #342 changed, it didn't change RB tree's allocation profile.

The cleanest read is "same workload, different host noise floor".
Apple M4 Pro thermals shift run-to-run; we did not pin CPU,
disable Spotlight, or quiesce the OS between sessions. Future
re-measurements should batch all three implementations within
the same /usr/bin/time wave to reduce drift.

#342's expected impact (per its own evidence: −3.6 pp record-leak
rate, +369 k record_frees, +45 k variant_frees) lives on workloads
heavy in record bindings flowing through `SLet` pass-through.
RB tree allocates only variants and boxed `Int`s in functions
without SLet-of-records pass-through, so the RB tree path simply
does not exercise the code #342 fixed. The +7-cell variant_frees
delta is consistent with that: tiny stdlib churn, nothing
RB-shaped.

## Gap decomposition revisited

#305 attributed the 9.7× / 41× gap to three structural issues.
For each, did the night's fixes (through #342) move the needle?

### Gap 1 — Spine rebuild (~22 variant allocs / insert)

**#305 hypothesis:** `balance(...)` rebuilds three new `RBNode`
cells per spine step; the shape predicate (issue #210, PR #289)
declines reuse on the four rotation arms because the rewrite
reorganises the tree shape, not just repaints. ~22 allocs/insert
× 1 M = 21.6 M variant allocs.

**Now:** `tag variant allocs = 21 612 396` — **byte-identical** to
#305. No PR in the merge window touched issue #210 or the four
rotation arms. **No movement, as predicted.**

### Gap 2 — Boxed `Int` parameters (~7 M extra allocs)

**#305 hypothesis:** every `k`, `v`, `n` parameter inside
`balance` / `insert_loop` heap-allocates a `kai_int` because
phase-2 unboxing (issue #267) only fires on local arithmetic
chains, not parameters. Issues #251 / #252 (parameter unboxing)
are post-MVP.

**Now:** `tag int allocs = 7 017 419` — **byte-identical** to #305.
#251/#252 didn't land in the window. **No movement, as predicted.**

### Gap 3 — Missing decrefs on dropped intermediate trees

**#305 hypothesis:** old spines aren't freed when `t` drops; we
see `alloc_total - free_total ≈ 21.96 M` cells leaking, equivalent
to ~96 bytes × 21.96 M ≈ 2.0 GB residual heap (matching the
1929 MB RSS within rounding).

**Now:** `alloc_total - free_total = 21 956 437` cells (vs #305's
21 956 434) — **+3 cells over 28.6 M, statistically zero**.
RSS = 1929 MB, exactly as in #305. Even with #342's SLet
pass-through fix landed, the RB tree leak is unmoved because
the leak comes from the spine rebuild path (Gap 1) holding old
trees alive through tail-recursive calls, not from SLet
record pass-through.

**No movement.**

### Summary of gap movement

| Gap                       | #305 contribution     | now            | Moved?  |
| ------------------------- | --------------------- | -------------- | ------- |
| 1 — Spine rebuild         | 21.6 M variant allocs | 21.6 M         | **No**  |
| 2 — Boxed Int params      | 7.02 M int allocs     | 7.02 M         | **No**  |
| 3 — Missing decrefs       | 21.96 M leak          | 21.96 M leak   | **No**  |

All three gaps are unchanged. The wall-clock variation across
sessions is normal noise; the ratios kaikai/C and kaikai/C++
drift but the structural counters do not.

## Why each delta happened (or didn't)

- **kaikai wall +14.6 % pre→post-#342 within this lane**: not a
  #342 regression. The RC counters changed by 8 cells; the
  workload's allocation profile is identical; #342's fix path
  isn't touched by RB tree. Most likely cause is between-session
  thermal / OS noise on the M4 Pro host. The post-#342 sample is
  4× tighter than the pre-#342 sample, so the post-#342 number
  is the more trustworthy median.
- **kaikai/C ratio drifted +23 %** vs #305: same noise story,
  amplified by C also dropping in absolute terms. The ratio is
  more sensitive to host state than the counters.
- **RSS unchanged**: the leak is structural (Gap 3), not RC
  discipline, so PRs that improved discipline elsewhere (#293,
  #294, #295, #342) don't help here.
- **Counters unchanged**: every PR in the merge window — #301,
  #304, #307, #313, #320, #324, #333, #334, #337, #339, #342 —
  touches code paths the RB tree workload doesn't exercise. The
  driver uses no closures over the hot loop, no strings beyond
  the final `print`, no nullary variants in the spine, no tail
  calls into multi-use parameters, and no SLet-of-records
  pass-through. None of those optimisations have a surface area
  on this benchmark.

## Conclusions vs Anga Roa DoD #4

DoD #4 target: kaikai/C ≤ 1.7× (stretch) or 2.0× (relaxed) wall.

- **2026-05-06 (PR #305, 0.43.0):** 9.68× wall, 41.0× RSS. Failed.
- **2026-05-08 (now, 0.44.1 post-#342):** 11.90× wall, 40.7× RSS.
  Failed.

DoD #4 status: **unchanged in substance**. The headline ratio
moved up by ~2× because of host noise on the wall-clock numerator
and denominator; the structural picture (counters, RSS, gap
decomposition) is byte-identical. Neither closer nor further on
the dimensions that matter for closing the gap.

The roadmap path forward is unchanged from #305:

1. **Tighten the reuse-in-place predicate** to admit
   constant-arity rotations (Gap 1). Drops variant allocs
   from 21.6 M to ~2 M.
2. **Parameter unboxing** (issues #251 / #252). Drops int
   allocs from 7 M to ~0.
3. **Per-call-site decref discipline in `balance` /
   `insert_loop`** (extension of #293/#294/#295/#342 to
   tail-recursive helpers without SLet pass-through). Drops the
   21.96 M leak to ~0.

Each of those is a load-bearing runtime / emitter lane, not a
benchmark-fixture patch. Filing the gaps as issues remains the
right next step.
