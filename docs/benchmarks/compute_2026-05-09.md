# Compute benchmarks — Phase 3 unboxing landed (2026-05-09)

This snapshot pins the wall-clock impact of **Phase 3 unboxing**
(issue #383) on the two compute-heavy benchmarks the lane was
chartered to close: `fib(35)` and Project Euler #4.

The pre-Phase-3 numbers come from the **post-Phase-2 v0.46.0**
release (tag `v0.46.0`, 2026-05-09 just before the lane opened);
the post-Phase-3 numbers come from this branch's selfhost build.

Hardware: Darwin 25.4.0 / Apple silicon, clang -O2 for the C
reference. Every cell is the **median wall-clock** of seven warm
runs; hot caches assumed.

## Headline

| Benchmark         | C -O2 | kaikai pre  | kaikai post | pre/C  | post/C  |
|-------------------|-------|-------------|-------------|--------|---------|
| `fib(35)`         | 18 ms | 338 ms      | **17.6 ms** | 18.8×  | **1.0×** |
| Euler #4          |  2 ms |  23 ms      | **2.0 ms**  |  9.5×  | **1.05×** |

Both numbers smash the lane's acceptance gate (≤ 3.3× and ≤ 5×
respectively). With Phase 3 the kaikai compiler emits raw C
arithmetic on raw scalars across function call boundaries; the
benchmark fixtures recurse through several user-defined fns
(`fib`, `reverse_int`, `is_palindrome`, `search`) without any
`KaiValue *` boxing in the hot path. clang -O2 then inlines the
recursion and the kaikai output collapses onto the same shape as
the C reference.

## Reproduce

```sh
make kaic2

clang -O2 examples/perceus/phase3/fib35_bench_c.c   -o /tmp/fib35_c
clang -O2 examples/perceus/phase3/euler4_bench_c.c  -o /tmp/euler4_c

bin/kai build examples/perceus/phase3/fib35_bench.kai   -o /tmp/fib35_kai
bin/kai build examples/perceus/phase3/euler4_bench.kai  -o /tmp/euler4_kai

# median wall-clock over 7 runs (Python `time.perf_counter` driver)
python3 -c '
import subprocess, time, statistics, sys
prog = sys.argv[1]
samples = []
for _ in range(7):
    t0 = time.perf_counter()
    subprocess.run([prog], stdout=subprocess.DEVNULL, check=True)
    samples.append((time.perf_counter() - t0) * 1000)
print(f"{prog}: {statistics.median(samples):.1f} ms")
' /tmp/fib35_kai
```

## Why the post numbers move so much

Pre-Phase-3 every recursive call paid:

1. `kai_int(...)` — heap allocation of a fresh boxed Int per arg.
2. `->as.i` — pointer chase into the box to pull the raw scalar
   back at the callee body's first arithmetic node.
3. `kai_decref(...)` — RC bookkeeping at the boundary.

Phase 2 had already collapsed the inner-loop arithmetic to raw
`int64_t`; the call boundary stayed boxed (Phase 2 non-goal #1).
Phase 3 closes that gap: every fn whose params + return are in
{Int, Bool, Char, Real} is classified as a UFn and emits the
raw C signature `static int64_t fib(int64_t kair_n)`. Recursive
sites pass raw scalars; the kaikai-side `kai_apply` indirection
only fires when the call sites reach for a closure or pass
through a generic.

## Where the gap remains

`fib(35)` and Euler #4 are the **best case** for Phase 3 — they
recurse through user fns whose entire signature is primitive and
the inner work is pure arithmetic. Programs that route through
generic functions, protocol dispatch, effect handler clauses, or
variant constructors still pay the boxed call-boundary cost
because those signatures cannot be lowered without
monomorphisation tweaks (post-1.0 work). The RB-tree benchmark,
for instance, sits behind variant-cell allocation (#384) plus
the Perceus reuse-in-place story; Phase 3 does not move that
needle on its own.
