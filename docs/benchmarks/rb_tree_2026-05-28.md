# RB-tree state-of-the-base — 2026-05-28 (pre-Koka-comparison baseline)

Fresh measurement of the canonical Perceus red-black tree workload
(`examples/perceus/rb_tree_bench.kai`, 1 M random inserts) to establish
an honest baseline *before* installing Koka and comparing against the
Perceus paper (Reinking, Xie, de Moura, Leijen, POPL'21, §7). All prior
docs in `docs/benchmarks/rb_tree_*.md` are from 2026-05-08/09 — three
weeks and several Perceus/unboxing lanes stale.

This is a **measurement doc**: no compiler or stdlib code was changed.

## Headline numbers (5 runs, median)

| Implementation | Wall (median) | RSS (peak) | vs C wall | vs C RSS |
| --- | ---: | ---: | ---: | ---: |
| **kaikai** | **3637 ms** | **294 MiB** | **14.78×** | **6.26×** |
| C (hand-written intrusive RB) | 246 ms | 47 MiB | 1.00× | 1.00× |
| C++ `std::set<int64_t>` | 235 ms | 47 MiB | 0.96× | 1.00× |

The Hanga Roa DoD #4 target is kaikai/C ≤ 1.7× (stretch) or 2.0×
(relaxed). We are at **14.78×** — far from target, and **essentially
unchanged from the 15.9× baseline of 2026-05-08** despite Phase 4
Option A (variant-field unboxing) shipping its layout on 2026-05-14.

### Caveat on the comparison's fairness

The C reference is an **intrusive, mutable, in-place** RB tree: it does
~1 `malloc` per *new* insert and mutates nodes via parent pointers, with
zero allocation during rebalancing. kaikai runs the **functional
Okasaki** algorithm: every insert rebuilds the root-to-leaf spine,
allocating ~22 fresh `RBNode` cells per insert. These are different
memory models. The whole point of Perceus reuse-in-place is to make the
functional rebuild *reuse the consumed cells* and approach the in-place
cost — so the 14.78× gap is precisely the gap that Perceus is supposed to
close, and the right yardstick is "how close does reuse-in-place get us,"
not "is functional == intrusive" (it cannot be without reuse).

## Two findings

### 1. Phase 4 Option A IS active — but the boxed frontier neutralises it

`RBNode(Color, RBTree, Int, Int, RBTree)` emits as
`kai_variant_u(14, "RBNode", 5, 80, ...)`. Mask 80 = the two `Int` slots
(positions 2 and 3) are unboxed `.i64` (`16 + 64`); the three pointer
slots are 0. So the **storage** of `k`/`v` is unboxed inside the node —
Phase 4 layout works as designed.

Yet the workload still does **68.75 M `int` allocations (75 % of all
allocs)** and the wall is unchanged from the pre-Phase-4 baseline. The
reason is in the emitted construction code:

```c
{.i64 = kai_take_int(kai_internal_dup(kai_lk))}
```

To place an `Int` field into the unboxed `.i64` slot, the emitter still
routes through a **boxed `KaiValue *`**: `kai_take_int` reads `v->as.i`
then `kai_decref(v)`. Every field read that crosses the boxed↔unboxed
frontier (construction, `kai_op_lt` on keys, passing `k`/`v` as boxed fn
args because `insert_loop`/`balance` have boxed signatures) allocates a
fresh `kai_int` box, extracts the scalar, and frees the box. The storage
is unboxed; the **frontier traffic is not**. This is exactly the
follow-up the honesty-targets named: *"primitive-slot extract that
survives without a fresh `kai_int` alloc."* Until that lands, Phase 4
Option A buys the smaller cell but not the alloc/RC reduction the
2026-05-09 breakdown predicted (~60 % RC traffic cut → not realised).

This is the kaikai-vs-Koka crux. In Koka a small `Int` is a value-
immediate (tagged pointer), never a heap box, so the field read is a
register move with no alloc and no RC. kaikai still boxes `Int` at every
function boundary because `RBNode` is not an all-primitive type, so the
Phase 3 signature-unboxing that made `fib`/Euler reach ~1× C does not
apply to the keys travelling alongside the `RBTree` pointer.

### 2. Allocs appear to have GROWN ~3.7× vs the 2026-05-09 breakdown (UNVERIFIED)

The 2026-05-09 breakdown measured **24.6 M allocs**; today both
`KAI_TRACE_RC` and `KAI_PROFILE_RC` agree on **91.4 M allocs** — a 3.7×
increase, with RC traffic roughly flat (803 M vs 860 M). The 05-09 doc
predates the Phase 4 layout ship (05-14).

**Hypothesis (NOT yet verified):** Phase 4 Option A, as implemented,
unboxed the *storage* but *multiplied the frontier `kai_int` temporaries*
— every `.i64` field that must cross back to boxed (for a comparison or a
boxed-signature call) now allocates a throwaway box, where before the
field shared one box with the node. If true, Phase 4 Option A in its
current shape **regressed the alloc count without improving the wall**,
and the missing "extract without fresh alloc" follow-up is not just a
perf nicety but a correction of a regression.

**To verify:** check out the pre-Phase-4 commit (layout shipped
2026-05-14), rebuild the same benchmark, and compare `alloc_total`. Left
for the next session; flagged here so the number is not taken as settled.

## reuse-in-place: still firing zero

The benchmark wrapper's note ("reuse-in-place fires, issue #210") is
stale. `KAI_TRACE_RC` reports no `reuse_in_place` count on this workload,
consistent with the honesty-targets' "self-compile firing remains 0."
The RB rebuild is the textbook reuse-in-place shape (constant-arity
`RBNode` rewrites on a uniquely-owned spine), so this is the single
biggest unrealised Perceus win: ~22 M variant allocs/frees per run that
reuse-in-place would collapse toward in-place mutation.

## What this baseline says for the Koka comparison

Three axes to measure Koka against, with kaikai's current standing:

1. **Wall: 14.78× C.** Koka's paper reports the functional RB tree
   within ~1.0-1.1× of its C++ baseline. The gap to explain is ~14×.
2. **RSS: 6.26× C (294 vs 47 MiB).** Perceus markets "garbage free" — Koka
   should be at or near 1× C RSS. kaikai's 6× is leaks (3.4 M live) +
   the functional rebuild's live-peak (4 M cells) + boxed-int overhead.
3. **Allocs: 91.4 M (vs ~1 M for in-place C).** The two levers that move
   this are (a) value-immediate `Int` (kill the 68.75 M int boxes) and
   (b) reuse-in-place (collapse the 22.6 M variant churn). Koka has both;
   kaikai has neither realised on this workload.

