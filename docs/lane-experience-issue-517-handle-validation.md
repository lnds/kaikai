# Lane experience — issue #517: handle block validation

**Lane**: `lane-issue-517-handle-validation`
**Closes**: [#517](https://github.com/lnds/kaikai/issues/517) — handle block: residual effects + clause shape not validated
**Predecessor**: #516 (general effect-row propagation through ordinary fn calls); same negative-space audit batch as #511.
**Date**: 2026-05-12.

## Scope as planned

Per the #511 silent-contract README, five fixtures lived in
`examples/negative/silent_contract/` documenting that `handle { body }
with Eff { ... }` accepted four distinct shapes of malformed input
plus one main-level row miss:

1. `handle_residual_effect.kai` — body performs `Net.fetch` (residual)
   while the handler only catches `Io`. Today: compiles + segfaults.
2. `handle_partial_with_other_effect.kai` — body performs `Io` and
   `Log`; handler catches `Io`. `Log` should surface to main's row.
3. `handle_clause_missing_resume.kai` — clause omits the trailing
   `resume` param. Today: silent + downstream C arity mismatch.
4. `handle_clause_wrong_arity.kai` — clause adds extra positional
   params before `resume`. Today: silent + ill-formed C.
5. `main_row_user_effect.kai` — `main : Int / Net` for a user-defined
   effect with no default handler. Today: silent + segfault.

Brief asked for: (A) diagnosis, (B) residual-effect typing fix,
(C) clause-shape + unknown-`with`-effect validation, (D) fixture
promotion, (E) tier0/tier1/selfhost gates, (F) retro + PR.

## Scope as shipped

All six asks landed. Three commits, no scope creep beyond the brief
except updating one existing positive fixture
(`examples/effects/op_call_basic.kai`) whose `main : Int / Io` row
exercised exactly the same shape the new validation rejects; the
fixture was reshaped to drive the same code path inside a `handle`
block so the smoke test remains representative.

## Where the gaps lived

### Part 1 — residual effects (typer)

The synth-time row arithmetic in `synth_handle`
(`stage2/compiler.kai:28433`) was correct already: body row is
walked under an empty row, the handled label is stripped with
`filter_labels_matching`, and the residual is merged back into the
prev_row before clauses are typed. The leak was downstream —
`check_body_row`'s REmpty branch had a `main`-only escape hatch that
absorbed every label except `Ffi` (issue #516 hardcoded that single
exception). Any user-defined effect (`Net`, `Log`, ...) or any
builtin effect lacking a runtime default handler reached `main` and
disappeared. The fix replaces `only_non_absorbed_labels`'s
`if h.eff == "Ffi"` literal with a predicate
`effect_has_default_handler(st, name)` that consults a closed list of
absorbable stdlib effects (the canonical `Stdout..Spawn` handlers
emitted by `default_shims_for`, plus the parametric stdlib effects
`State`/`Reader`/`Writer` whose handlers are user-installed or
generated by the `var x = init` desugar).

### Part 2 — clause arity (typer)

`check_clauses_types_loop` already looked up each clause's op scheme
via `bind_clause_env_typed`, but the inner `bind_params_with_sig_typed`
fell back to lenient param binding (`bind_params_lenient`) on any
length mismatch, recording no error. Added `check_clause_arity` as a
pre-step that compares `list_length(params)` to
`arg_tys.length + 1` (the `+1` is `resume`) and emits a dedicated
diagnostic. The check is gated on a known op scheme so unknown ops
keep their existing `unknown op X on effect Y` diagnostic from
`check_clause_names` instead of stacking two errors.

### Part 3 — unknown `with Eff` (typer)

Added `check_handle_effect_known` before `check_clause_names` to
surface "unknown effect in `with`: `<Eff>`" eagerly. The legacy
behavior (one diagnostic per clause that all read "unknown op X on
effect <Eff>") is preserved as a cascade; the new error fires first
so the user sees the structural problem before the per-clause
fallout.

### Part 4 — main row vs handler set (typer)

The RClosed branch of `check_body_row` previously enforced only the
body-vs-declared subset relation. Added `check_main_row_handlable`
to walk `st.row` for `main` and emit a typed-hole-style diagnostic
naming the canonical handler set whenever a declared label has no
runtime evidence path. `Ffi` is excepted (capability tag — declaring
it in main's row is exactly how a program earns the right to call
extern fns); `State`/`Reader`/`Writer` are also excepted (handlers
are user/desugar-supplied). Everything else — user-defined effects
and unaccounted builtins — is rejected with concrete advice.

## Design decisions and alternatives

### Absorption set vs reject-everything-not-default-handler

The first draft rejected at main any label not in the
`default_shims_for` list (Stdout..Spawn + Ffi as REmpty-only excludes).
Re-running tier0 caught two regressions:

1. The `demos/stack/main.kai` demo uses `var xs = []` inside its
   `with Stack { ... }` handler, which the parser lifts into a
   wrapping `with State[List[Int]](init)` handle. The typer's
   `filter_labels_matching` is shallow (uses `ty_unifiable_shallow`
   on already-substituted types) and cannot bind the `?fresh`
   element type of `[]` against `List[Int]` produced by the inner
   clauses' `xs := ...` ops. The State label phantom-leaks past its
   own wrapping handle, reaches main, and the strict check rejects.
2. `examples/effects/ffi_extern_c_basic.kai` declares
   `main : Int / Ffi` and calls a `puts` extern. With the strict
   check, the explicitly-declared `Ffi` row label is rejected even
   though declaring it is the exact prescribed way to authorise FFI.

The pragmatic answer: keep the absorption set wide for stdlib
effects (everything injected by `inject_builtin_effects` minus
`Ffi`), and treat `Ffi` as REmpty-rejecting but RClosed-accepting.
Rejecting all phantom State leaks is the principled long-term fix
but is out of scope for #517; it requires reworking
`labels_args_unify` to be a real unifier that extends the substitution.

### One up-front "unknown with-effect" diagnostic vs strict mode

Considered making `check_handle_effect_known` short-circuit the
remaining handle validation (no clause typing if the effect is
unknown). Rejected — the cascade is informative ("here's where each
clause body went wrong") and matches how the rest of the typer
handles errors: report and continue.

### Where to place `main_row_user_effect.kai`

Considered `pub_effect/` and a new `main_row/` subdir. Settled on
`handle_leak/` to keep all five #517 fixtures in one place and avoid
fragmenting the negative-space layout for a one-shot issue.

## Structural surprises

1. The `check_body_row` `Ffi`-only escape hatch was designed for one
   capability label and grew by accretion. Replacing the literal
   with a predicate exposed the parametric-stdlib-effect gap above —
   not a new bug, just a row-arithmetic limitation that was
   previously hidden by silently absorbing everything.
2. `op_call_basic.kai` is the only positive smoke test whose
   `main : Int / Io` row uses a user-defined `Io` — every other
   effects-positive fixture either wraps in `handle` or uses the
   builtin `Io` alias. The fixture was reshaped, not deleted.
3. `effect_default_compatible(decls, name)` already exists at emit
   time for issue #308 (user shadowing of canonical decls), but
   `check_body_row` does not have `decls` in scope. Added a
   parallel env-based predicate `user_shadows_default_eff` that
   asks the type env whether the canonical ops of a shadowable
   builtin are all present.

## Coverage gaps left for next lanes

1. **Parametric-effect row arithmetic**: `filter_labels_matching` is
   shallow and cannot strip a parametric label when the wrapping
   handler's `eff_tyargs` were inferred from a polymorphic init
   (`[]` typed as `List[?fresh]`). The State/Reader/Writer absorption
   set papers over this; the principled fix is a real unifying
   filter and a `check_main_row_handlable` that runs against the
   substituted row. Tracked separately.
2. **`with Eff` arity validation**: `with State` without ty_args
   (when an init type can be inferred) is accepted; `with State[Int, Int]`
   is silently truncated. Doesn't touch the silent contracts from
   #511 — saved for a future audit pass.
3. **Resume-position-by-name validation**: the brief asked to check
   the resume param is correctly named/positioned per language
   convention. Current state checks COUNT only — the trailing param's
   name is convention but not enforced. A clause like
   `print_io(s, r) -> r(())` works today and reads naturally; making
   "the trailing param must be named `resume`" a hard error would
   regress the alias case. Left as written documentation.

## Real cost vs estimate

Estimate from the brief: 2-3 days.
Actual: one session (~3 hours active diagnosis + implementation +
two rebuild cycles to chase the State demo and Ffi-row regressions).
The diagnostic-first approach (read `synth_handle` end-to-end,
trace each fixture by hand against the code, draft the predicate
before coding) saved the longer-tail rebuild churn that the bug-bash
batch retros warned about.

## Selfhost

Byte-identical. Tier0's selfhost gate is the load-bearing check
here; the compiler.kai uses `handle { ... } with Mutable { ... }`
in a handful of places (the per-frame perceus rewrite and the
arena allocator's `var` use sites), all of which pass the new
arity + known-effect checks without modification.
