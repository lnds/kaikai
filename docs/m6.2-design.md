# m6.2 — Qualified module calls

Status: design draft 2026-04-27. Pre-implementation.

## Goal

Make `import list` introduce the namespace `list`, so that
`list.map(xs, f)` is a callable form distinct from any unqualified
`map` and from a record-field projection on a value named `list`.
Selective `import list.{map}` brings `map` into the unqualified
scope while leaving the rest under the prefix. `import list as L`
binds `L.map`. Resolver, typer, and codegen treat `module.fn` as a
single name lookup in the module's export table, not a record-field
projection on a value named `module`.

Prerequisite for **m14** (naming-convention migration: `list_map` →
`list.map`). Without m6.2 the rename is a parser error.

## Scope (in)

- `module.fn(args)` resolves to the function `fn` exported by module
  `module`. Distinct from any unqualified `fn`. Distinct from a
  record-field projection.
- `import path.{f, g}` brings `f`, `g` into the unqualified scope
  (today's M6.1 behaviour) **and** keeps them accessible via
  `path.f`, `path.g`. Both forms callable.
- `import path as Alias` binds `Alias.f`, `Alias.g`. The plain path
  segment (`path`) does not become a binding.
- Two modules that export the same function name no longer clash —
  `kai_<module>__<name>` minting keeps them distinct in the C ABI.
- M6.1 fixtures keep working unchanged (every existing `import path`
  call site that uses unqualified names continues to resolve).

## Scope (out)

- **Re-exports**. `pub use path.{f, g}` is m6.3 territory.
- **Visibility scoping**. `pub fn` already exists; private fns
  remain unexported the same as today.
- **Cyclic module imports**. Already handled by M6.1's visited set.
- **Module-level state / `effect` ops in modules other than the
  importing file**. Those work today through M6.1's concat and stay
  working — the qualified surface adds, does not replace.
- **The naming-convention migration itself** (m14). m6.2 only makes
  the rename mechanically possible; the rename pass is m14's job.

## The disambiguation question

The parser produces `EField(lhs, name)` indistinguishably for:

- `value.field` — record projection on a runtime value
- `module.fn` — qualified call to an exported function

The decision must be made by **the resolver / typer**, not the
parser. Three plausible rules; the design adopts (B):

| Rule | When `EField(EVar(n), m)` resolves as a module call |
|------|-----------------------------------------------------|
| (A) | When `n` is in the module table, **always**. Locals named like a module become unreachable via dot syntax. |
| (B) | When `n` is in the module table **and** `n` is not a local binding in scope. Locals shadow imports. |
| (C) | When `n.m` matches a `(module, exported_name)` pair. If both fail, fall through to record-field projection. |

(B) matches Rust / Python / OCaml: a `let list = ...` in scope
shadows `import list`. A diagnostic warns about the shadow but the
program type-checks. (A) is simpler but surprising — a local
binding that happens to share a name with an import is silently
unreachable through dot syntax. (C) is the most permissive but
makes type errors more confusing because `value.bad_field` could
be reported as "module `value` does not export `bad_field`" when
the user intended a record projection.

We adopt **(B)**. The resolver tracks two scopes: the local
lexical scope (already exists) and the module table (new). The
field-access typer first asks: "is `obj` an `EVar(n)`, with `n`
unbound locally and bound in the module table?" If yes, dispatch
qualified. Otherwise, fall through to today's record-projection
logic.

## Architecture

Five components touched. Order matches the pipeline.

### 1. Module table — populated during `expand_imports`

A new state pair carried alongside `ResolveState`:

```kai
type ModuleTable = { entries: [ModuleEntry] }
type ModuleEntry = { name: String, exports: [String], origin: String }
```

- `name` — the import surface name (`list` for `import list`,
  `helpers` for `import shared.helpers`, the alias for
  `import path as Alias`).
- `exports` — the names of the public decls in the module file
  (functions + types + effect declarations). Selective imports
  pre-filter this list.
- `origin` — the absolute file path; used for diagnostics ("module
  `list` defined at `stdlib/core/list.kai`").

`expand_imports` walks each `DImport` and, in addition to
concatenating decls (which keeps M6.1 working), records a
`ModuleEntry` per import. For `IkSelective(p, names)`, the
`exports` field is `names`; the rest of the module's decls remain
in the global concat stream but are not reachable via the prefix
form.

For `IkAlias(p, alias)`, the `name` field is `alias` and `exports`
covers every public decl in the module. The unqualified surface
form is **not** populated for aliases (you wrote `as Alias`
precisely to keep the bare names out of scope), so the
concat-into-global step skips alias imports.

For plain `IkPath(p)`, today's concat behaviour is preserved
(every public decl lands in unqualified scope) **and** the
`ModuleEntry` is recorded (every public decl is also reachable via
the prefix). Both surfaces work; the user can pick.

### 2. Resolver / `Env` — local scope unchanged; module lookup added

The lexical environment (`Env { scopes, ... }`) keeps its current
shape. A separate `module_table` is threaded through the resolver
state. When the resolver sees `EField(EVar(n), m)`:

1. If `n` appears in any local scope of `Env`, leave the
   `EField` alone (record projection path). This is the
   shadow-rule from option (B).
2. Otherwise, look up `n` in `module_table`. If `n` is a module
   and `m` is in its `exports`, rewrite the AST node to
   `EModCall(n, m)` (new variant — see §4). The downstream typer
   and codegen handle that variant; the original `EField` shape
   is reserved for record projection.
3. Otherwise, leave the `EField` alone. The typer's record-field
   logic will report the appropriate error (and crucially, the
   error message can hint "did you mean to import `n`?" when
   `n` is a known module that the user did not import).

### 3. AST — new `EModCall` variant

```kai
type ExprKind
  | ...
  | EField(Expr, String)
  | EModCall(String, String)   # m6.2: module-qualified function reference
  | ...
```

`EModCall(mod, fn)` is the resolved form of a qualified call's
**callee**. It appears only inside an `ECall` head position;
standalone `EModCall` (i.e. taking the module function as a
first-class value) is allowed and emits the same C symbol as a
regular `EFn` thunk would, just under the qualified name.

Why a new variant rather than mutating `EField`: `EField` shows
up in many code paths today (record projection, struct emission,
op-call rewrites for effects). Reusing it for module dispatch
forces every consumer to dispatch on a runtime tag ("is the LHS
a module?"). A separate variant lets each pass match exactly one
of the two cases and keeps the code paths textually parallel to
how they read today.

### 4. Typer — `EModCall` lookup

The typer treats `EModCall(mod, fn)` as a regular function
reference whose type is the scheme of the function exported by
`mod` named `fn`. The lookup is:

1. From `module_table`, find the entry for `mod`.
2. From the entry, find the `Decl` for `fn` (already in the
   concatenated `decls` list, but indexed by module via
   `module_table`).
3. Use the existing `fn_scheme_of_decl` machinery to produce the
   scheme.

`synth_field` (line 15364) is **not** modified — it still handles
`EField` for record projection only. The resolver pass (§2)
guarantees that no `EField(EVar(mod), fn)` reaches `synth_field`
when `mod` is a module not shadowed locally.

### 5. Codegen — `kai_<module>__<name>` minting

The C symbol minting changes for functions that come from
imported modules (vs. functions defined in the user's top-level
file or in the prelude):

- Function decls in the user's top-level file: `kai_<name>` (today).
- Prelude builtins: `kai_prelude_<name>` (today).
- Functions imported from a module: `kai_<module>__<name>` (new).
  - Two underscores between module and name to avoid clashing
    with valid identifier characters in either side.

This change has **bootstrap-impact**: every kaic2 self-compile
emits new symbol names for stdlib calls. The migration plan
(see §"Migration") staggers the rename so selfhost stays viable.

`emit_ident_value` learns to look up the function's module of
origin (via `EFn` extended with an optional `module: Option[String]`
field) and mint the appropriate symbol. `emit_call_expr` for an
`EModCall(mod, fn)` head emits `kai_<mod>__<fn>` directly.

## Migration

The naming change is the riskiest part. Strategy:

1. **m6.2 v1** (this lane): both surfaces work. The prefixed
   minting (`kai_<module>__<name>`) lands but is **opt-in** —
   only triggered when the user writes `module.fn` (qualified
   call) or `module as Alias`. Plain `import module` keeps
   minting `kai_<name>` for backward compat.
2. **m6.2 v2** (follow-up): plain `import module` mints prefixed
   too, so `import list` + `import string` + their respective
   `map` exports stop clashing. Diagnostic added: "module `list`
   exports `map` and module `string` exports `map`; use
   qualified form to disambiguate".
3. **m14**: stdlib's flat-prefix names (`list_map`, `string_split`)
   retire. The functions inside `stdlib/core/list.kai` are
   declared as `pub fn map(xs, f)` (no prefix); callers reach
   them as `list.map`.

## Test strategy

New fixtures under `examples/modules-qualified/` exercise:

- `qualified_basic.kai` — `import list` + `list.map(...)` round-trip.
- `qualified_with_alias.kai` — `import list as L` + `L.map(...)`.
- `qualified_selective.kai` — `import list.{map}` + both
  `map(...)` (unqualified) and `list.map(...)` (qualified).
- `qualified_local_shadow.kai` — `let list = ...` in scope after
  `import list`; verify the local wins for dot syntax.
- `qualified_two_modules_clash.kai` — `import list` + `import string`
  both exporting `map`, callable distinctly via prefix.
- `qualified_record_projection_still_works.kai` — `let p =
  Point { x: 1, y: 2 }; let x = p.x` keeps working (no
  regression on record projection).

Negative fixtures:

- `qualified_unimported_module.err.expected` — `random.gen()`
  without `import random` errors with "module `random` is not
  imported; did you forget `import random`?"
- `qualified_unknown_export.err.expected` — `list.nope()` errors
  with "module `list` does not export `nope`; available
  exports: map, filter, ..."

## Decisions (confirmed 2026-04-27)

1. **Disambiguation rule**: **(B) — locals shadow imports**, per
   Rust / Python / OCaml. A `let list = ...` in scope after
   `import list` makes the local win for dot syntax. The
   resolver checks the lexical environment first; only if no
   local match, the module table is consulted.
2. **C symbol naming**: **`kai_<module>__<name>`** (double
   underscore separator). Single underscore was rejected because
   it clashes with valid Kai identifiers (e.g., `list_map`).
3. **Migration**: **two sub-phases**. v1 lands the prefixed
   minting **opt-in** (only when the user writes `module.fn` or
   `import path as Alias`). Plain `import path` keeps minting
   `kai_<name>` and stays compat with M6.1. v2 (separate lane)
   makes the prefixed minting universal and adds the
   "two modules export the same name" diagnostic.
4. **`pub use` (re-exports)**: deferred to **m6.3**. Out of scope
   for this milestone.
5. **Fixture layout**: **`examples/modules-qualified/*`**
   (separate directory). Matches the precedent of
   `examples/modules-path/` — each module-system feature has its
   own directory, target keeps simple iteration semantics, no
   mixing of helper files with top-level fixtures.
