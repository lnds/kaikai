# m6.2 v2 — Universal prefixed C-symbol minting

Status: **LANDED 2026-04-28**. Lane: `m6.2-v2-minting`. See
`CHANGELOG.md` § `[0.5.0]` for the implementation summary and the
test-plan delta.

Follow-up to `docs/m6.2-design.md` v1. Implements the prefix policy
that v1 staged behind an opt-in flag (only `module.fn` qualified
calls emitted prefixed; plain `import module` kept the legacy
`kai_<name>` to stay compat with M6.1).

## Goal

Every public decl owned by an imported module mints `kai_<module>__<name>`
(double underscore separator). Decls owned by the root file (the user
program) keep `kai_<name>`. This unblocks:

  - `m14 v1.B-E`: rename `pub fn opt_map` → `pub fn map` in option.kai
    while result.kai also exports `map` — both land on distinct C
    symbols (`kai_option__map`, `kai_result__map`).
  - User-side shadow of prelude: `fn sum` in a demo no longer collides
    with `pub fn sum` from `stdlib/core/list.kai` (`kai_list__sum`).
    Demos that today fail to link (euler1 `fn sum`, quicksort `fn sort`)
    start working without source edits.

## Design

### Decl ownership, captured at expand time

Add a 10th field to `DFn`:

```kai
DFn(Bool, String, [String], [Param], Option[TypeExpr], RowExpr, Expr,
    Int, Int,
    Option[String])   # module_origin
#                       ^^^^^^^^^^^^^^^^^^^^
#                       None  = root-file decl (kai_<name>)
#                       Some(m) = imported from module m (kai_<m>__<name>)
```

`expand_imports` and the prelude loader populate the field at the point
of concat:

  - `resolve_module(...)` already calls `non_import_decls(decls)` and
    appends them to the flat stream. Wrap that step in a "tag every
    DFn with `module_origin = Some(<surface_name>)`" pass before the
    append.
  - `load_preludes` (m14 v0) similarly tags decls with the prelude
    file's basename minus `.kai`.

Other `Decl` variants (`DType`, `DEffect`, `DProtocol`, `DUnit`,
`DTest`, `DImport`, `DImpl`, `DDerive`, `DAxiom`, `DUse`) do **not**
need the field — they don't generate user-callable C symbols, or
their generated symbols are already qualified (`__pimpl_<P>_<T>_<op>`).

### EFn carries the origin

`EFn(String, Int, Option[String])` — name, arity, module_origin.
`register_fn_decls` reads the origin from the new DFn field.

### Codegen rules

The C symbol minted for a function reference (decl or call):

```
sym(name, mod_origin) = match mod_origin {
  None -> "kai_" ++ name
  Some(m) -> "kai_" ++ m ++ "__" ++ name
}
```

Three sites mint the symbol:

  1. **Decl emission** (`emit_fn` / `emit_decl_header` / `_kai_<name>_thunk`).
     Uses the decl's own `module_origin`.
  2. **Call emission** (`emit_named_call_lookup`, `emit_pipe_named_lookup`,
     `emit_call_expr` `EVar` / `EModCall` branches). Looks up the
     callee in `[EFn]`, uses the matched entry's `module_origin`.
  3. **Value emission** (`emit_ident_value` for first-class fn refs).
     Same lookup, same prefix.

The `EModCall(mod, fname)` AST variant already carries `mod` — codegen
mints `kai_<mod>__<fname>` directly without consulting `[EFn]`.

### Use-side disambiguation

When a bare `EVar(name)` resolves to multiple entries in `[EFn]` with
distinct `module_origin`s:

  - If exactly one has `module_origin = None` (root file), prefer it.
    This matches the universal "locals shadow imports" rule applied
    at decl scope.
  - Otherwise, error: "ambiguous reference to `name`: comes from
    `<mod1>` and `<mod2>`. Use `<mod1>.<name>` or `<mod2>.<name>` to
    disambiguate."

The diagnostic fires lazily — only when user code actually uses the
ambiguous name, not at registration time. Two unused exports with
the same name across modules are legal as long as no caller writes
the bare form.

### Stage 0 + stage 1: no change

Stage 0's emit.c keeps minting `kai_prelude_*` for runtime builtins
and is not touched. Stage 1's `compiler.kai` keeps minting `kai_<name>`
flat — stage 1's selfhost path (`make -C stage1 selfhost`) compiles
`stage1/compiler.kai` without preludes, so `module_origin` is
universally `None` and the output is unchanged. Stage 1 also compiles
`stage2/compiler.kai` (during the `kaic1 → kaic2` build), and that
input also doesn't pull preludes — every decl's origin is the root
file. No backport required.

The change activates only when `bin/kai` (or the demo / test
makefiles) pass `--prelude` flags; that is a stage-2-only path.

## Migration impact

  - Stage 2 `make selfhost` byte-identical: **OK** by construction
    (selfhost target uses no preludes; module_origin is None for
    every decl; minting is unchanged).
  - Stage 2 `make selfhost-llvm` byte-identical: **OK** by the same
    argument.
  - Demos / examples / `bin/kai`-built programs: **stdlib symbols
    rename**. `kai_list_take` → `kai_list__list_take`,
    `kai_opt_map` → `kai_option__opt_map`, etc. Every legacy alias
    `pub fn list_X = X(...)` (29 of them in list.kai post-v1.A) keeps
    working as a callable surface — they are just renamed in the C
    output.
  - `kai_prelude_*` runtime builtins: **unchanged**. Those are not
    declared in `.kai` source; they live in `stage0/runtime.h` and
    are reached via the `prelude_table()` lookup.

## Phases

  1. Decl-side: add `module_origin` to `DFn`; thread `expand_imports`
     + prelude loader; mint prefixed symbols in C + LLVM emit.
  2. Use-side: extend `EFn`; update the four call/value emission
     paths; update `EModCall` branches to use `mod` directly.
  3. Ambiguity diagnostic at bare-name lookup.
  4. Stage 0 + stage 1: no change. (Validated by survey, not a phase
     of work.)
  5. Selfhost convergence + regression test (two preludes export
     same name, no link collision).
  6. Restore demos: re-validate `euler1` + `quicksort` (locals
     shadow prelude). If clean, add to baseline.

## Open questions resolved during survey

  - **Decl shape vs side-table?** Decl shape (extra field on DFn).
    A side-table indexed by name fails on the case the lane exists
    to fix (two `pub fn map` decls in the flat stream) — name alone
    does not disambiguate. Carrying origin on the decl itself is the
    only sound shape.
  - **Prefer "None" or first-defined on bare-name ambiguity?**
    Prefer `None` (root file). Matches the rule a Python / Rust /
    OCaml programmer expects: a top-level local definition silently
    takes precedence over an import. Across-module ambiguity (two
    Some(m) entries) is the actual hazard and stays an error.
  - **Module name normalization?** Use the surface name a user wrote
    on the import line, which is what `ModuleEntry.name` already
    holds (`last_segment` of the dotted path, the alias from
    `IkAlias`, or the basename from prelude loader). No further
    mangling — module names are already valid C identifier prefixes.
