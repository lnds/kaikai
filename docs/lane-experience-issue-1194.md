# Lane experience — issue #1194 (reassigned record var field read in interpolation)

## Scope as planned vs shipped

**Planned.** Fix the typer regression where a reassigned record-typed
`var` whose field is read inside a `#{...}` interpolation fails with
"type annotation needed: the receiver of `.f` is not resolved". Sibling
of #1186 (same error string, same `drain_one_deferred_field` arm) but a
distinct cause. Add a per-PR regression fixture; confirm the kaikai-vs-rust
case 1 (`trace=1 2 1 2 1 2`) recompiles.

**Shipped.** That, plus a second coupled defect the brief's root-cause
direction did not anticipate. The issue was bisected and diagnosed by the
integrator; the lane went straight to the fix. No scope creep beyond the
second defect, which the same repro surfaced once the first was fixed.

## Root cause — two coupled defects, not one

The brief's hypothesis was that the interpolation desugar loses the
receiver type of the lifted field access (a structural loss on the lift).
AST dumps disproved that: the desugar output for the failing case was
**byte-identical** to a compiling case (`cur.get().id` lifted correctly).
The failure was elsewhere, and there were two.

**Defect 1 — op-label dedup drops a payload tyvar without pinning it.**
A `var cur := a; cur := a` desugars to a canonical `handle State[P] as cur`
with `get` and `set` ops. In the body, `cur.set(a)` infers first: its
payload is concrete, so the row carries `State@cur[P]`. Then `cur.get()`
infers: its return is a fresh tyvar `?r`, label `State@cur[?r]`.
`st_add_label`'s dedup uses `label_present_unifiable`, which treats a free
tyvar as unifiable but is a **pure shape check** — it never extends the
substitution. It sees `State@cur[?r]` "unifies" with `State@cur[P]`, drops
the `get`'s label, and `?r` stays free. The field access `cur.get().id`
then drains against a free receiver → the error. Without the reassignment
there is no concrete `set` label to dedup against, so `?r` resolves via the
handler init — which is exactly why the no-reassign case compiled.

The deferred-field drain tries to rescue `?r` via
`pin_free_receiver_from_decl`, but that pins from the **declared** row of
the enclosing fn (`main : Int / Stdout`), which never mentions `State` —
the handler is internal, generated by the `var` desugar. So the rescue
cannot fire.

Fix: unify the label ty_args pairwise before the dedup drops one
(`st_unify_dedup_label`, called from `synth_op_call_with_scheme_keys`),
**gated on the `@alias` cell tag**. The first cut gated only on the
shallow-unifiable check and regressed `dual_actor_request_reply` (#472):
that fixture declares two *independent* instances `Actor[Request]` +
`Actor[Reply]` in one row, which key by concrete `name[args]` and carry
NO alias. The shallow check declared `Actor[Reply]` unifiable with a
still-free `Actor[Request]` arg and cross-bound them → `type mismatch in
op call Actor.send`. A design consult (asu) had flagged exactly this: the
shallow-unifiable check is too lax a precondition for forcing unification
in general; it is sound only when instance identity lives in the label.
For cells that identity is the `@alias` (from #802's cell-identity work);
a bare effect has none. So the fix threads ty_args only between labels
carrying `@alias` — cells, whose ops all share one state type by handler
construction. Multi-instance aliasless effects are left untouched.

**Defect 2 — interp span scan off-by-one skips adjacent segments.** The
full kaikai-vs-rust case 1 (`trace := "#{trace}#{int_to_string(cur.id)} "`)
hit a *different* error after defect 1 was fixed: `cannot find cur in
scope`. Cause: `desugar_var_block` chooses the fast array-slot lowering
for `cur` unless `body_escapes` sees a naked cell read inside an interp
span (`interp_span_reads_cap`). That scan's `icap_expr_end` returned the
index PAST the closing `}` instead of the brace itself, so the caller's
`expr_end + 1` overshot by one. For two *adjacent* `#{...}#{...}` regions
the overshoot landed on the second region's `{`, skipping its `#{` opener
— so the naked `cur` there was never seen, `cur` specialised to an array
slot, and the interp-lifted bare `cur` referenced a name no longer in
scope. A single char between segments (a space) masked the bug; only true
adjacency triggered it. Fix: mirror `iscan_expr_end` in fnreg.kai — the
depth-1 closing brace returns its own index.

## Structural surprises the brief did not anticipate

- The desugar lift is **not** where the receiver was lost. The brief
  pointed at `desugar_interp_decls` / the lift threading the receiver type;
  the AST dump showed the lift is correct and the real fault was in
  `st_add_label` (infer) and `icap_expr_end` (a scan helper, not the lift).
  Reading the actual post-desugar AST before trusting the root-cause
  direction saved a wrong fix.
- The two defects are **sequential**: the second is only reachable once
  the first is fixed and the field resolves far enough to reveal the naked
  cell read. Fixing only defect 1 leaves case 1 broken with a different
  error.
- The `icap_*` scan claimed in its comment to "mirror `iscan_expr_end`"
  but did not — a copy that drifted from its source. The mirror was
  restored, not reinvented.

(+`.out.expected`), wired as `test-issue-1194-...` into `.PHONY`,
