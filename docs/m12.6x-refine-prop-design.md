# m12.6.x Lane — Refinement type propagation (#1 + #2)

Status: design draft 2026-04-28. Pre-implementation.

## Goal

Make refinement types (`Int where >= 0`, `String where matches ~r/.../`) carry **static guarantees** through the typer instead of behaving as a runtime-only assert. Closes the third leg of "audit-trail-compile-time" (effects + units + refinements) by making the type system *use* the predicates the parser already attaches.

Today (post-m12.6.a–e):
- Parser produces `TyRefine(base, pred)` from `BaseT where Pred`.
- Resolver lifts to `TyRefineT(base, pred)` semantic type.
- Unifier treats refinement as **transparent**: `TyRefineT(b, p) ⊑ b` and `b ⊑ TyRefineT(b, p)` both pass without consulting `p`.
- Runtime: `requires` / `ensures` desugar to `assert`s. `where` clauses on parameters are dropped at semantic level (predicate stays in syntactic AST for diagnostics only).

What changes after this lane:
- **#1**: unifier rejects downcasts in argument position. Calling `f(x: Int where >= 0)` with `x: Int` errors at compile time unless the caller can prove `>= 0` (literal, refined-arg pass-through, narrowed by surrounding control flow).
- **#2**: a small interval-propagation pass infers bounds for `Int` (and bool literals) and constant-folds predicate evaluations when the bounds are tight enough to decide the predicate.

The two are coupled: #1 enforcement is only useful once #2 can prove the obvious cases without the user manually casting.

## Scope (in)

- `Int` refinements with comparators (`<`, `<=`, `==`, `!=`, `>`, `>=`) against literals: `where >= 0`, `where 1 <= self <= 65535`.
- `Bool` refinements: `where self` / `where not self`.
- Static check at call sites: literal arguments evaluated at compile time, refined-source arguments forwarded by subsumption.
- Const propagation: `let n = 5; f(n)` where `f` requires `>= 0` — n inferred as `Int where == 5`, subsumes.
- Diagnostic: refinement violations report the failing predicate + the inferred bounds (when known).

## Scope (out)

- **String regex refinements** (`where matches ~r/.../`) — depend on regex stdlib (Lane B). Dropped from this lane; left as runtime check.
- **Path-dependent refinements** (`where x > y` for two vars). Single-variable predicates only.
- **SMT-grade reasoning**. The interval evaluator is a few hundred lines max; if a predicate cannot be decided at compile time, it falls back to runtime assert (current behaviour).
- **Fractional intervals**. Real refinements stay runtime-only in v1.
- **Disjunction**. `where self == 1 or self == 2` falls back to runtime.
- **Match-arm narrowing for refinements**. Already partially landed (#3 v1, commit `d2a61c4`); kept where it stands.

## Architecture

### #1 — Refinement-aware unify

Today's unifier collapses `TyRefineT(b, p)` to `b` in both directions. The new form:

```kai
unify(actual, expected, s) :
  match (actual, expected):
    (a, TyRefineT(eb, p)) ->
      # Caller wants refined; must verify actual satisfies p.
      let s1 = unify(a, eb, s)
      let known = predicate_holds(actual_origin(a), p, s1)
      match known {
        Provably(true)  -> Some(s1)
        Provably(false) -> None  -- reject at compile time
        Unknown         -> Some(s1)  -- defer to runtime assert
      }
    (TyRefineT(ab, p), e) ->
      # Refined value flowing into unrefined slot — always OK.
      unify(ab, e, s)
    -- ... rest unchanged
```

The `Unknown` branch preserves today's behaviour (runtime assert), so any program that compiles today keeps compiling. Only **provably-false** cases become compile errors.

`predicate_holds` consults a small interval table built by #2.

### #2 — Interval propagation

A second-pass walker over the typed AST that, for each `Int`-typed `Expr`, attaches a derived interval `[lo, hi] | None`:

| Node | Interval |
|---|---|
| `EInt(n)` | `[n, n]` |
| `EVar(x)` | typed-env lookup; if `Int where p`, derive from `p`; else None |
| `EBinop("+", a, b)` | `[lo_a + lo_b, hi_a + hi_b]` (saturating on overflow) |
| `EBinop("-", a, b)` | `[lo_a - hi_b, hi_a - lo_b]` |
| `EBinop("*", a, b)` | conservative: take min/max of cross product |
| `EIf(c, t, e)` | join of `t`'s and `e`'s intervals |
| Otherwise | None |

Refinement predicate evaluation given an interval `[lo, hi]`:

| Predicate | Provably true if | Provably false if |
|---|---|---|
| `self >= k` | `lo >= k` | `hi < k` |
| `self <= k` | `hi <= k` | `lo > k` |
| `lo' <= self <= hi'` | `lo >= lo' and hi <= hi'` | disjoint |
| `not p` | `p` provably false | `p` provably true |
| `p1 and p2` | both true | either false |
| `p1 or p2` | either true | both false |
| Otherwise | Unknown | Unknown |

This evaluator is `try_eval_pred` (m12.6.d) extended to take an interval table instead of only constant literals.

### Pipeline insertion

The interval pass runs **after** typing finalises, but **before** the assert-emission pass. It mutates a side-table `interval_env: [(ExprId, Interval)]` consumed by:
1. The unifier when checking `TyRefineT` constraints (Provably / Unknown).
2. The assert-emission pass to **drop** runtime checks that are now provably true.

If the pass fails to compute an interval, we silently degrade to runtime assert. This keeps the lane backward-compatible.

## Phases

1. **Research & design doc** (this) — done.
2. **#2 interval propagation pass** — pure new code, doesn't alter unifier yet. Computes intervals, doesn't enforce. Validate against fixtures that print inferred intervals (debug-only).
3. **#1 unifier reshape** — now consults the interval table. New "provably-false" diagnostics. Existing fixtures must continue to pass.
4. **Assert dropping** — predicates that compile-time-decided are removed from emit. Selfhost re-validates.
5. **Fixtures + docs** — positive/negative end-to-end tests, update `m12.6` doc to mark #1 + #2 as landed.

## Risks

- **Selfhost stability**: `compiler.kai` itself uses `Int` heavily. Any miscomputed interval risks breaking valid code. Mitigation: degrade silently to "Unknown" on any uncertainty; never reject compile-time without high confidence.
- **Interval blow-up on multiplication**: pessimistic intervals can over-approximate quickly. Cap intervals at `Int` range; treat unbounded as None.
- **Interaction with units of measure**: `Decimal<USD> where >= 0` — interval propagation must traverse `TyDimT` transparently.
- **Compile-time perf**: interval pass is O(#typed_nodes). Not a concern for typical programs but watch the selfhost wall-clock budget.

## Open questions

1. Should the `Provably(false)` case be a hard error or a warning under a flag (e.g. `--strict-refinements`)? Hard error means existing tests that rely on the runtime check might break. Recommend: hard error, with a fix-it suggestion in the diagnostic.
2. Should we ship narrowing on `if x > 0 { ... }` so the then-branch sees `x: Int where > 0`? That is part of #3 v2, **not** this lane. Skip for now.
3. Bool refinements scope: include or defer? Recommend: include (cheap to add, useful for `where self`).
