(`/tmp/kaikai-portfolio-demo/portfolio.kai`) using `--prelude
stdlib/protocols.kai`. None of these block Core; all of them are
real and should be fixed for Full or earlier.

## Bug 1 — Nested derive runtime panic

**Status: Fixed (incidentally) in m12.8.x phase 2.**

The phase-2 typed-AST finalisation pass that fixes Bug 2 also
resolves the unresolved-TyVar problem inside synthesised
`#derive(Show)` impl bodies. After phase 2, the post-inference
protocol-call rewrite picks the concrete `__pimpl_Show_<Inner>_show`
direct call. Fixture: `examples/protocols/m12_8_y_nested_derive.kai`.

**Symptom (pre-fix)**: `panic: __protocol_dispatch_Show_show` at
runtime when calling `show(record)` where the record has a sum-type
field that also uses `#derive(Show)`.

**Repro**:
```kai
#derive(Show)
type Inner = A | B | C

#derive(Show)
type Outer = { tag: Inner, n: Int }

fn main() : Int {
  let o = Outer { tag: A, n: 42 }
  print(show(o))   # panicked pre-phase 2; now prints "Outer { tag: A, n: 42 }"
  0
}
```

**Original root-cause hypothesis**: the post-inference protocol-call
rewrite resolved `show(o.tag)` only at the outermost call site;
the nested `show(...)` synthesised inside the auto-generated
`impl Show for Outer` body didn't see the same context because the
typer hadn't pinned the field type when the rewrite walker read it.

**Actual fix**: the typed-AST finalisation pass walks every
`Expr.ty` annotation and applies the per-decl substitution after
inference completes. Synthesised impl bodies' `EVar` reads then
carry concrete head types at rewrite time.

## Bug 2 — Protocol call inside higher-order lambda panics

**Status: Fixed in m12.8.x phase 2** (`stage2/compiler.kai`
`finalise_typed_expr` walker, called from `infer_decl`).

**Symptom (pre-fix)**: same panic message, triggered inside a
lambda passed to `each` / `map` / `filter`.

**Repro**:
```kai
type Tx = { n: Int }
impl Show for Tx { fn show(t: Tx) : String = int_to_string(t.n) }

fn main() : Int {
  let xs : [Tx] = [Tx { n: 1 }, Tx { n: 2 }]
  each(xs, (t) => print(show(t)))   # panicked pre-phase 2
  0
}
```

**Root cause confirmed**: `synth_lambda` annotated the lambda's body
expressions with TyVars at the moment it walked the body. When the
outer `ECall(each, [xs, lambda])` later unified, the substitution
mapped `?t2 := Tx`, but no pass re-applied that substitution to the
saved `Expr.ty` annotations inside the lambda body. The post-
inference protocol-call rewrite read `t.ty = Some(TyVarT(2))`,
`ty_head_name` returned `None`, and the rewrite skipped — leaving
the dispatcher panic in the emitted code.

**Fix**: a finalisation pass at the end of `infer_decl` walks the
typed body and applies `apply_ty(st2.sub, t)` to every `Expr.ty`.
After this, the rewrite reads concrete head names and produces
direct `__pimpl_<P>_<T>_<op>` calls. Fixture:
`examples/protocols/m12_8_y_lambda_dispatch.kai`.

## Bug 3 — LLVM backend prints `Real<unit>` as `0`

**Status: Fixed in m12.8.x phase 2** (`stage2/compiler.kai`
LLVM `EReal(v)` arm).

**Symptom (pre-fix)**: `real_to_string` (or `show` for `Real`)
returned `"0"` for any `Real<u>` value when compiled with
`--emit=llvm`. The C backend printed the actual value.

**Root cause confirmed**: the LLVM emitter's `EReal` arm hard-coded
`call %KaiValue* @kaix_real(double 0.0)` — a placeholder from the
M12 round-trip era when the compiler itself contained no live real
literals. User code with `100.0<USD>` and friends needs the real
lexeme to round-trip.

**Fix**: emit `kaix_real(double <real_to_string(v)>)`, with a `.0`
suffix appended when `real_to_string` returns an integer-valued
double (LLVM IR rejects `double 100`; `double 100.0` is required).
Fixture: `examples/protocols/m12_8_y_llvm_real_unit.kai`.

## Bug 4 — `#{x}` interpolation does not dispatch via `Show`

**Status: Fixed in m12.8.x phase 2** (`stage2/compiler.kai`
`desugar_interp_decls` pass).

**Symptom (pre-fix)**: `"...#{x}..."` always used the runtime
universal printer (`kai_to_string`), even when `T` had `impl Show`.
The user's custom `show(x)` was silently bypassed.

**Repro**:
```kai
#derive(Show)
type Money = { amount: Int, currency: String }
impl Show for Money {
  fn show(m: Money) : String = int_to_string(m.amount) ++ " " ++ m.currency
}

fn main() : Int {
  let m = Money { amount: 100, currency: "USD" }
  print("Balance: #{m}")          # pre-fix: "Balance: {amount: 100, currency: USD}"
  print("Balance: " ++ show(m))   # always: "Balance: 100 USD"
  0
}
```

**Root cause confirmed**: the interp body inside an `EStr` span was
parsed at C-emit time with `parser_new("<interp>", expr_src, ...)`
and routed through `kai_to_string(...)`. The expression never
passed through the typer or the post-inference protocol-call
rewrite, so no `Show` impl was ever consulted.

**Fix**: a post-`lower_protocols` desugar pass `desugar_interp_decls`
walks every `EStr` node, parses each `#{...}` body up-front, and
rewrites the whole string into
`string_concat_all([..., __proto_show(parsed_expr), ...])`. The
typer infers types for the lifted parts, the finalisation pass
resolves them, and the rewrite routes each `__proto_show` call to
the matching `__pimpl_Show_<T>_show`.

The desugar is conditional on `Show` being declared in the program
(the registry's `op_names` contains `"show"`). Files that use
`#{int_to_string(x)}` without ever declaring/loading `Show`
continue through the legacy emit-time `kai_to_string` path
unchanged. Fixture: `examples/protocols/m12_8_y_interp_show.kai`.

## Gap 1 — No `impl Show for Real<u: Unit>` in stdlib

**Status: Partial fix in m12.8.x phase 2.**

**Shipped**: a compile-time reflection intrinsic
`unit_name(x: Real<u>) : String` that returns the canonical unit
symbol of its argument (extracted from the typed `.ty` slot at
emit time). Both backends lower the call to a constant string
literal; the runtime sees no special machinery. With this
intrinsic users can write
```kai
print("#{x} #{unit_name(x)}")     # → "100 USD"
```
to combine the numeric value (rendered via `impl Show for Real`) and
the unit symbol without ever calling `unitless`.

**Deferred**: the original ask was a parametric
`impl Show for Real[u: Unit]` so that a bare `show(x)` (or `#{x}`)
on `Real<USD>` would automatically include the unit. That requires
per-unit specialisation in the protocol-impl machinery — currently
`proto_type_name(TyDim(base, _))` collapses to the base
constructor's name, so all `Real<u>` for different `u` dispatch to
the same impl fn whose body has no concrete `u` to feed
`unit_name`. Lifting this constraint would touch the impl
registry, the dispatcher generator, and the rewrite step.
Estimated >1 day of structural refactor; explicitly out of phase-2
budget.

**Suggested follow-up lane (call it m12.8.z)**: extend the impl
table key from `(P, T_head)` to `(P, T_head, T_args)` so a
parametric impl `Show for Real[u: Unit]` can monomorphise per
unit. The dispatcher generator would emit one impl per
instantiation seen at call sites; the rewrite would key on the
same tuple. Stage-1 round-trip cost should be small (only
proto_reg storage grows).

Repro for the deferred case:
```kai
unit USD
fn main() : Int {
  let x : Real<USD> = 100.0<USD>
  print(show(x))         # current: "100" (dispatches via impl Show for Real)
                         # ideal:   "100 USD" (would need the deferred impl)
  print("#{x} #{unit_name(x)}")  # available now: "100 USD"
  0
}
```

## Bug 5 — `show(x)` inside `impl Show for X` looks recursive

**Status: Fixed in m12.8.x phase 2** (`stage2/compiler.kai`
`desugar_interp_kind` ECall arm + `validate_resolved_protocols`
post-rewrite walker).

**Symptom (pre-fix)**: ergonomic, not functional. Inside an impl
body the nested protocol calls dispatch to *other* impls, but
visually they looked like recursive calls into the function being
defined.

**Repro**:
```kai
impl Show for Tx {
  fn show(t: Tx) : String =
    show(t.kind) ++ "  " ++ show(t.amount)
  # the two `show(...)` look like recursive calls; they actually
  # resolve to __pimpl_Show_TxKind_show / __pimpl_Show_Real_show.
}
```

**Fix**: ship the postfix method-call sugar `x.show()` that
desugars to `show(x)`. Visually unambiguous that the dispatch is
on the receiver, not the function being defined:
```kai
impl Show for Tx {
  fn show(t: Tx) : String =
    t.kind.show() ++ "  " ++ t.amount.show()
}
```

The desugar lives in the existing post-`lower_protocols` pass: when
an `ECall(EField(receiver, op_name), args)` is encountered and
`op_name` is a registered protocol op, rewrite to `ECall(EVar(
__proto_<op>), [receiver, ...args])`. The post-inference rewrite
then routes to the matching `__pimpl_<P>_<T>_<op>` direct call.

**Companion fix — compile-time missing-impl diagnostic**:
`validate_resolved_protocols` walks the typed AST after the rewrite
and reports any `__proto_<op>` call whose receiver type is concrete
but has no impl. The diagnostic names the protocol, the type, and
the operation. This closes a v1 limitation (the bare runtime
panic) and is what the `m12_8_y_postfix_no_impl` negative fixture
expects.

Records that happen to carry a field named the same as a protocol
op shadow the new sugar — same edge as the existing `__proto_<op>`
rename pass; users hit it iff they declare a field whose name is a
protocol op, which is unusual. Fixtures:
`examples/protocols/m12_8_y_postfix_method_call.kai` and
`examples/protocols/m12_8_y_postfix_no_impl.kai`.

## Composite consequence (post-phase 2)

The demo's balance line, originally:
```kai
print(show(unitless(balance(txs))) ++ " USD")
```
can now be written as either of:
```kai
print("Balance: #{balance(txs)} USD")                         # hard-code unit
print("Balance: #{balance(txs)} #{unit_name(balance(txs))}")  # via reflection
```
Both produce identical output on C and LLVM backends.

The original spec-intended form
```kai
print("Balance: #{balance(txs)}")
```
remains pending Gap 1's deferred parametric impl.
