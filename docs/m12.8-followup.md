# m12.8 follow-up: known v1 bugs and gaps

Issues found while writing a 70-line external demo
(`/tmp/kaikai-portfolio-demo/portfolio.kai`) using `--prelude
stdlib/protocols.kai`. None of these block Core; all of them are
real and should be fixed for Full or earlier.

Generated 2026-04-26 from empirical demo, post-m12.8 + m12.5 + m5.x
+ m7e13 landed.

**Status (m12.8 phase 2 ŌĆö Core REOPENED, 2026-04-27):** Bugs 1-5 and
Gap 1 **Fully Fixed**. The earlier 2026-04-26 partial close on Gap 1
landed the `unit_name` reflection intrinsic; the parametric
`impl[u: Unit] Show for Real<u>` form now ships too. The parser
accepts impl-level tparams (`impl[u: Unit] P for T<u> { ... }`),
each method's tparams are prefixed with the impl-level binders so
the typer treats the body as a regular generic fn, and a call-site
rewrite (`try_rewrite_show_dim_real`) synthesises the unit-aware
expression `string_concat(real_to_string(__strip_unit(x)),
" <UNIT>")` whenever the argument resolves to a concrete unit. The
synthesis sidesteps the still-identity m4c monomorphiser ŌĆö the
parametric body's `unit_name(x)` would otherwise emit "" because
the body's `x.ty` keeps the impl-level UVar after inference. Bare
`Real` falls through to the parametric body which returns just
`real_to_string(__strip_unit(x))`. `__strip_unit` is the
typer-internal erasure intrinsic exposed for this purpose only
(NOT user-callable; see `compiler.kai` near the
`prelude_names()` entry).

## Bug 1 ŌĆö Nested derive runtime panic

**Status: Fixed (incidentally) in m12.8.x phase 2.**

The phase-2 typed-AST finalisation pass that fixes Bug 2 also
resolves the unresolved-TyVar problem inside synthesised
`#derive(Show)` impl bodies. After phase 2, the post-inference
protocol-call rewrite picks the concrete `__pimpl_Show_<Inner>_show`
direct call. Fixture: `examples/protocols/m12_8_y_nested_derive.kai`.

**Symptom (pre-fix)**: `panic: __protocol_dispatch_Show_show` at
runtime when calling `show(record)` where the record has a sum-type
field that also uses `#derive(Show)`.

**Repro**:
```kai
#derive(Show)
type Inner = A | B | C

#derive(Show)
type Outer = { tag: Inner, n: Int }

fn main() : Int {
  let o = Outer { tag: A, n: 42 }
  print(show(o))   # panicked pre-phase 2; now prints "Outer { tag: A, n: 42 }"
  0
}
```

**Original root-cause hypothesis**: the post-inference protocol-call
rewrite resolved `show(o.tag)` only at the outermost call site;
the nested `show(...)` synthesised inside the auto-generated
`impl Show for Outer` body didn't see the same context because the
typer hadn't pinned the field type when the rewrite walker read it.

**Actual fix**: the typed-AST finalisation pass walks every
`Expr.ty` annotation and applies the per-decl substitution after
inference completes. Synthesised impl bodies' `EVar` reads then
carry concrete head types at rewrite time.

## Bug 2 ŌĆö Protocol call inside higher-order lambda panics

**Status: Fixed in m12.8.x phase 2** (`stage2/compiler.kai`
`finalise_typed_expr` walker, called from `infer_decl`).

**Symptom (pre-fix)**: same panic message, triggered inside a
lambda passed to `each` / `map` / `filter`.

**Repro**:
```kai
type Tx = { n: Int }
impl Show for Tx { fn show(t: Tx) : String = int_to_string(t.n) }

fn main() : Int {
  let xs : [Tx] = [Tx { n: 1 }, Tx { n: 2 }]
  each(xs, (t) => print(show(t)))   # panicked pre-phase 2
  0
}
```

**Root cause confirmed**: `synth_lambda` annotated the lambda's body
expressions with TyVars at the moment it walked the body. When the
outer `ECall(each, [xs, lambda])` later unified, the substitution
mapped `?t2 := Tx`, but no pass re-applied that substitution to the
saved `Expr.ty` annotations inside the lambda body. The post-
inference protocol-call rewrite read `t.ty = Some(TyVarT(2))`,
`ty_head_name` returned `None`, and the rewrite skipped ŌĆö leaving
the dispatcher panic in the emitted code.

**Fix**: a finalisation pass at the end of `infer_decl` walks the
typed body and applies `apply_ty(st2.sub, t)` to every `Expr.ty`.
After this, the rewrite reads concrete head names and produces
direct `__pimpl_<P>_<T>_<op>` calls. Fixture:
`examples/protocols/m12_8_y_lambda_dispatch.kai`.

## Bug 3 ŌĆö LLVM backend prints `Real<unit>` as `0`

**Status: Fixed in m12.8.x phase 2** (`stage2/compiler.kai`
LLVM `EReal(v)` arm).

**Symptom (pre-fix)**: `real_to_string` (or `show` for `Real`)
returned `"0"` for any `Real<u>` value when compiled with
`--emit=llvm`. The C backend printed the actual value.

**Root cause confirmed**: the LLVM emitter's `EReal` arm hard-coded
`call %KaiValue* @kaix_real(double 0.0)` ŌĆö a placeholder from the
M12 round-trip era when the compiler itself contained no live real
literals. User code with `100.0<USD>` and friends needs the real
lexeme to round-trip.

**Fix**: emit `kaix_real(double <real_to_string(v)>)`, with a `.0`
suffix appended when `real_to_string` returns an integer-valued
double (LLVM IR rejects `double 100`; `double 100.0` is required).
Fixture: `examples/protocols/m12_8_y_llvm_real_unit.kai`.

## Bug 4 ŌĆö `#{x}` interpolation does not dispatch via `Show`

**Status: Fixed in m12.8.x phase 2** (`stage2/compiler.kai`
`desugar_interp_decls` pass).

**Symptom (pre-fix)**: `"...#{x}..."` always used the runtime
universal printer (`kai_to_string`), even when `T` had `impl Show`.
The user's custom `show(x)` was silently bypassed.

**Repro**:
```kai
#derive(Show)
type Money = { amount: Int, currency: String }
impl Show for Money {
  fn show(m: Money) : String = int_to_string(m.amount) ++ " " ++ m.currency
}

fn main() : Int {
  let m = Money { amount: 100, currency: "USD" }
  print("Balance: #{m}")          # pre-fix: "Balance: {amount: 100, currency: USD}"
  print("Balance: " ++ show(m))   # always: "Balance: 100 USD"
  0
}
```

**Root cause confirmed**: the interp body inside an `EStr` span was
parsed at C-emit time with `parser_new("<interp>", expr_src, ...)`
and routed through `kai_to_string(...)`. The expression never
passed through the typer or the post-inference protocol-call
rewrite, so no `Show` impl was ever consulted.

**Fix**: a post-`lower_protocols` desugar pass `desugar_interp_decls`
walks every `EStr` node, parses each `#{...}` body up-front, and
rewrites the whole string into
`string_concat_all([..., __proto_show(parsed_expr), ...])`. The
typer infers types for the lifted parts, the finalisation pass
resolves them, and the rewrite routes each `__proto_show` call to
the matching `__pimpl_Show_<T>_show`.

The desugar is conditional on `Show` being declared in the program
(the registry's `op_names` contains `"show"`). Files that use
`#{int_to_string(x)}` without ever declaring/loading `Show`
continue through the legacy emit-time `kai_to_string` path
unchanged. Fixture: `examples/protocols/m12_8_y_interp_show.kai`.

## Gap 1 ŌĆö No `impl Show for Real<u: Unit>` in stdlib

**Status: Fully fixed in m12.8 phase 2 (Core REOPENED, 2026-04-27).**

**Shipped (m12.8.x phase 2, 2026-04-26)**: the `unit_name(x: Real<u>) :
String` reflection intrinsic that returns the canonical unit symbol of
its argument (extracted from the typed `.ty` slot at emit time). Both
backends lower the call to a constant string literal.

**Shipped (m12.8 phase 2, 2026-04-27)**: a parametric
`impl[u: Unit] Show for Real<u>` form. The parser accepts impl-level
tparams (`impl[u: Unit] P for T<u> { ... }` ŌĆö the brackets immediately
follow `impl`, mirroring `fn[T1, T2]`). Each method's tparams are
prefixed with the impl-level binders so the typer sees `fn show[u: Unit]
(x: Real<u>) : String` and the body's `x` is properly unit-polymorphic.

The m4c monomorphiser is still identity (`monomorphise(tp) = tp.decls`,
deferred per m4c #1), so a single `__pimpl_Show_Real_show` body is
emitted with the impl-level UVar still unbound ŌĆö `unit_name(x)` inside
it evaluates to "" at C-emit time. The fix is a call-site rewrite in
`try_rewrite_proto_call`: when the dispatched op is `show` and the
argument resolves to `TyDimT(TyReal, concrete-unit)`, the rewrite
synthesises
```
string_concat(real_to_string(__strip_unit(x)), " <UNIT>")
```
inline, baking the unit symbol into the result via the same
`unit_name_of_ty` helper the existing `unit_name` intrinsic uses.
Bare `Real` (and `Real<1>`) skip the rewrite and fall through to the
parametric body, which returns `real_to_string(__strip_unit(x))` ŌĆö the
identity emit case for `__strip_unit` makes that equivalent to
`real_to_string(x)`.

`__strip_unit : Real<u> -> Real` is the typer-internal erasure
intrinsic introduced for this purpose. It reuses the C / LLVM emit
identity that the retired `unitless` user-visible name once held
(see Bug 8). Its `__` prefix marks it as a compiler-generated name ŌĆö
NOT user-callable; declaring it from user code would require adding
it to `prelude_names()`'s open list, which the v1 stdlib explicitly
does not do.

Fixtures:
- `examples/protocols/m12_8_phase2_show_real_unit.kai` ŌĆö single unit.
- `examples/protocols/m12_8_phase2_show_real_compound.kai` ŌĆö compound
  (`Real<EUR/USD>`) renders the canonical `EUR * 1 / USD` form
  produced by `unit_expr_display`.
- `examples/protocols/m12_8_phase2_show_real_bare_and_unit.kai` ŌĆö
  both `Real<USD>` and bare `Real` in one program; the rewrite only
  fires for the unit-bearing case.

Working repro:
```kai
unit USD
fn main() : Int {
  let x : Real<USD> = 100.0<USD>
  print(show(x))   # ŌåÆ "100 USD"
  print("#{x}")    # ŌåÆ "100 USD"
  0
}
```

The "extend the impl table to `(P, T_head, T_args)`" follow-up
proposed for the original m12.8.z lane is no longer needed for
Show; if Eq / Hash / Ord ever grow unit-aware semantics they would
follow the same call-site-rewrite pattern (their current bare impls
already work by dropping the unit, which is the right semantics).

## Bug 5 ŌĆö `show(x)` inside `impl Show for X` looks recursive

**Status: Fixed in m12.8.x phase 2** (`stage2/compiler.kai`
`desugar_interp_kind` ECall arm + `validate_resolved_protocols`
post-rewrite walker).

**Symptom (pre-fix)**: ergonomic, not functional. Inside an impl
body the nested protocol calls dispatch to *other* impls, but
visually they looked like recursive calls into the function being
defined.

**Repro**:
```kai
impl Show for Tx {
  fn show(t: Tx) : String =
    show(t.kind) ++ "  " ++ show(t.amount)
  # the two `show(...)` look like recursive calls; they actually
  # resolve to __pimpl_Show_TxKind_show / __pimpl_Show_Real_show.
}
```

**Fix**: ship the postfix method-call sugar `x.show()` that
desugars to `show(x)`. Visually unambiguous that the dispatch is
on the receiver, not the function being defined:
```kai
impl Show for Tx {
  fn show(t: Tx) : String =
    t.kind.show() ++ "  " ++ t.amount.show()
}
```

The desugar lives in the existing post-`lower_protocols` pass: when
an `ECall(EField(receiver, op_name), args)` is encountered and
`op_name` is a registered protocol op, rewrite to `ECall(EVar(
__proto_<op>), [receiver, ...args])`. The post-inference rewrite
then routes to the matching `__pimpl_<P>_<T>_<op>` direct call.

**Companion fix ŌĆö compile-time missing-impl diagnostic**:
`validate_resolved_protocols` walks the typed AST after the rewrite
and reports any `__proto_<op>` call whose receiver type is concrete
but has no impl. The diagnostic names the protocol, the type, and
the operation. This closes a v1 limitation (the bare runtime
panic) and is what the `m12_8_y_postfix_no_impl` negative fixture
expects.

Records that happen to carry a field named the same as a protocol
op shadow the new sugar ŌĆö same edge as the existing `__proto_<op>`
rename pass; users hit it iff they declare a field whose name is a
protocol op, which is unusual. Fixtures:
`examples/protocols/m12_8_y_postfix_method_call.kai` and
`examples/protocols/m12_8_y_postfix_no_impl.kai`.

## Bug 6 (Phase 4) ŌĆö Effects-prelude gap: prelude IO evades row enforcement

**Status: Fixed (partial) in m12.8.x phase 4 retake (a251689).**

The first phase-4 attempt halted because adding `/ Console` to
`stage2/compiler.kai`'s 39 print/eprint helpers broke the
stage-1 ŌåÆ stage-2 bootstrap (stage 1's parser predates the effect
system and rejected row syntax). The retake landed three pieces:

1. **Stage 1 parser tolerance** (~25 LOC). `parse_fn_decl` now
   consumes an optional `/ Eff (+ Eff)*` after the return type
   and discards the tokens up to the next `=` or `{`. Stage 1
   has no effect system semantically ŌĆö it just parses past the
   row ŌĆö but the tolerance lets stage 2's modified source
   round-trip through the bootstrap unchanged.

2. **Stage 2 typer enforcement**. `synth_call` calls
   `add_prelude_effect_label`, which adds `Console`/`Stdin`/
   `File` to the caller's row whenever the callee is a
   syntactically-bare `print`/`eprint`/`read_line`/
   `read_file`/`write_file`. The existing `check_body_row`
   subset check in `infer_decl` then rejects fns that perform
   these effects without declaring the matching row. Effect
   names follow Doc B's catalog (Console combines stdout +
   stderr; Stdout/Stderr split is deferred ŌĆö see phase 4b
   notes below).

3. **Builtin-effect injection scope** broadened.
   `inject_builtin_effects` now scans every fn's declared row
   (via `collect_used_row_labels`), not just `main`'s, so a
   helper declaring `: Unit / Console` triggers the Console
   builtin injection even when `main` doesn't mention it. Plus
   `each` / `map` / `filter` / `reduce` got row-polymorphic
   prelude signatures (`scheme_full([..], [0],
   TyFnT([..., TyFnT([..], _, row_with_tail(0))], _,
   row_with_tail(0)))`) so `xs |> each(dump_handler)` against
   a row-bearing callback typechecks. `synth_pipe` mirrors
   `synth_call`'s open-row trick.

39 fns in `stage2/compiler.kai` got `/ Console` (or
`/ Console + File`) added; 41 example fixtures gained the
appropriate row on `main` or helper signatures. `make
selfhost` / `make -C stage2 selfhost-llvm` / `make -C stage2
test-llvm-coverage` all still report 61 pass / 0 DIFF / 13
skip ŌĆö no parity loss from the enforcement.

New fixtures `examples/effects/m12_8_y_phase4_*`:

- `main_implicit` / `helper_explicit`: positives, both backends.
- `subtyping`: alias `type Io = Console + Stdin + Env + File`
  used as a wider declaration than the body's Console-only use.
- `no_row_negative`: `pure_fn(x) : Int` calling print is
  rejected with `effect not handled: Console`.

External demo `/tmp/kaikai-portfolio-demo/usd_to_eur.kai`
adapted (rows declared explicitly, Console used in place of
the deferred Stdout split) and runs identically on C and LLVM
backends with input `100\n0.92\n` producing the expected `92
EUR`.

### What didn't ship in this lane

**Main row inference**. A structural walker
(`prelude_eff_walk_*`) was prototyped to feed
`inject_builtin_effects` the right default handlers when main
has no declared row. It triggered runtime panics on certain
block shapes (`fn main() : Int { let r = 42; assert r == 42; r
}`) and after several bisect attempts the root cause stayed
elusive within the lane budget. The walker is reverted; every
fixture's `main` now declares its row explicitly. A future lane
can re-attempt main inference using a different scan strategy
(e.g. via the existing `fv_expr` walker post-resolution).

**Stdout / Stderr split**. The Doc B-style atomic catalog
keeps `Console` as the combined effect that owns both `print`
and `eprint`. Splitting requires a new pair of effect
declarations (each with its own op), runtime-side handler
struct splits, and a migration of every existing `Console.print
` / `Console.eprint` site. Out of scope; documented in
`stdlib/effects.kai`'s comment-only catalog as the target
shape.

> **Update 2026-04-27 ŌĆö Stdout / Stderr split shipped.** Phase 4b
> (`64b85fd`, `8b134bb`, `54d5364`, `ee49db2`) declared `Stdout` and
> `Stderr` as atomic per-stream effects, kept `Console = Stdout +
> Stderr + Stdin` as an alias for over-declaration, and added
> alias-aware op resolution so `Console.print(s)` rewrites to
> `Stdout.print(s)` at parse time. The Phase 3 demos
> (`examples/portfolio`, `examples/usd_to_eur`) consume the split
> directly; the dead `Console`-only scaffolding was removed in the
> Linus-review cleanup (`ee49db2`).

**`use Effect` open scope**. Custom effects' op calls still
need the `Eff.op(...)` prefix. The sugar prelude (`print(s)` Ōēł
`Console.print(s)`) is the only "open scope" that ships;
generalising it to user-declared effects is also deferred.

The phase-4 retake satisfies the lane's Tier 1 framing
(prelude IO now forces row declaration) without claiming the
remaining ergonomic improvements. Follow-up lanes can pick the
deferrals individually. Lane experience cross-ref:
`docs/lane-experience-m12.8.x.md` ┬¦"Phase 4 retake".

### Update 2026-04-27 ŌĆö `main` exception restored (49b4bd7)

The retake's first form over-applied enforcement: `check_body_row`
rejected `fn main() { print(...) }` whenever `main`'s row was
`REmpty`, conflicting with the pinned "Row obligatoria public +
private, **main excepci├│n ├║nica**" decision. Stage-0 fixtures
(`examples/minimal/{capture,hello,interp}.kai`) cannot carry an
effect row ŌĆö stage 0's parser predates the effect system ŌĆö yet stage
2 was rejecting them on the shared test harness path.

The fix in `49b4bd7` adds one branch to `check_body_row`: when the
declared row is `REmpty` AND the function name is `main`, the body's
performed effects are accepted unconditionally. Helpers still get
the strict subset check, so Bug 6 enforcement still bites where it
matters. Side adjustments in the same commit:

- `examples/minimal/{capture,hello,interp}.kai` reverted to
  `fn main() { ... }` (no row), restoring shared-harness compatibility.
- `stdlib/core.kai`'s `println : Unit` widened to `Unit / Console`.
- `examples/protocols/m12_8_x_derive_hash_sum_consistent.kai`'s
  `check` helper widened to `: Unit / Console`.
- `examples/protocols/m12_8_y_llvm_real_unit.{kai,out.expected}`
  deleted: depended on `unitless()` which Bug 8 (`f0acda2`) removed.
  The original Bug 3 (LLVM Real<unit>) fix lives in the compiler;
  the regression fixture was the only collateral.

`f8a2881` extended the same alignment to m12.8.z's lane during its
rebase: `validate_derive_ord_one`'s row widened to `: Int / Console`,
and the consistency-check fixture's helpers got `/ Console`.

A future lane that revives main row inference (the deferred
`prelude_eff_walk_*` walker) can replace this exception with the
inferred row; until then, declaring main's row explicitly remains
the recommended style.

## Bug 8 ŌĆö `unitless` leaks typer-internal erasure as user-facing fn

**Symptom**: `unitless[u: Unit](x: Real<u>) : Real` is exposed in the
prelude as if it were an ordinary library function, but it is
actually implemented as an emit-time identity special case (no
`kai_prelude_unitless` runtime function exists). Side effects of
the leak:

1. **Cannot be used as a first-class value**: `let f = unitless`,
   `xs |> unitless`, `map(xs, unitless)` all fail at C compile time
   with `use of undeclared identifier 'kai_unitless'`.
2. **Conceptual confusion**: a user reading the prelude sees
   `unitless` as a "library helper" comparable to `int_to_string`,
   when it is actually a typer-level erasure operation.
3. **Encourages bad patterns**: users reach for `unitless(x) ++ "
   USD"` when the design-intended idiom is `print("#{x}")` once
   `Show for Real[u: Unit]` lands (Gap 1).

**Repro**:
```kai
unit USD
fn main() : Int {
  let p : Real<USD> = 100.0<USD>
  let f = unitless              # error at C compile: kai_unitless undefined
  print(real_to_string(f(p)))
  0
}
```

**Root cause**: m12.5 implementation chose to elide the runtime
function ("`unitless()` runtime representation: implemented as
identity special-case in both C and LLVM emitters ŌĆö no
`kai_prelude_unitless` runtime function. Saves a runtime call and
matches the codegen-erasure invariant exactly.") The choice is
sound for codegen efficiency but the function should not have
been listed in the prelude as a user-callable value.

**Status**: **Fixed** in `f0acda2` (foreground, not via lane). The
fix removed `unitless` from `prelude_names` and the TyEntry table;
kept the emit special cases as dead code; deleted the two affected
fixtures (`m12_5_unitless_escape.kai`, `m12_5_print_real_unit.kai`);
added negative fixture `m12_5_unitless_unavailable.kai`. Stage 2
self-host stayed green, confirming the compiler does not use
`unitless` internally ŌĆö it was pure user-space leak.

## Possible audit (not urgent, post-Core)

Other prelude entries that may be removable as redundant ergonomic
sugar (NOT leaks like Bug 8):

- `string_concat(a, b)` ŌĆö `a ++ b` (m7d ┬¦23) covers this.
- `string_concat_all(xs)` ŌĆö `string_join("", xs)` covers this.
- `array_make` / `array_get` / `array_set` / `array_grow` ŌĆö `a[i]`
  and `a[i] := v` (m7b ┬¦6) cover the access patterns; the
  constructor (`array_make`) might still be needed.
- `int_to_string` / `real_to_string` ŌĆö if `Show.show(x)` is
  ergonomic enough post-Bug-4 and Gap-1, these become obsolete at
  user-call sites (still needed inside `impl Show`).
- `string_to_int` / `string_to_real` ŌĆö analogous, replaced by
  `Serialize.from_string(s)` in the Serialize protocol once
  return-type-driven dispatch lands.

These are **ergonomic tightening**, not bug fixes. Worth a
post-Core audit lane.

## Composite consequence (post-phase 2)

The demo's balance line, originally:
```kai
print(show(unitless(balance(txs))) ++ " USD")
```
now reduces to its spec-intended form
```kai
print("Balance: #{balance(txs)}")
```
which produces identical output on C and LLVM backends. The earlier
two-step workarounds
```kai
print("Balance: #{balance(txs)} USD")                         # hard-code unit
print("Balance: #{balance(txs)} #{unit_name(balance(txs))}")  # via reflection
```
still work ŌĆö the parametric impl is purely additive; existing fixtures
that already supplied the unit explicitly continue to compile and
run unchanged (none of them inject the suffix via Show, so no
double-suffixing). The old `m12_8_y_show_real_unit.kai` fixture was
the one place that did, and it has been migrated to the parametric
form (its expected output is unchanged).

## Post-Core REOPEN (2026-04-27, later same day) ŌĆö four bugs in `demos/*` cleanup

**Status: Core REOPENED** (see `docs/stage2-design.md` for the full
note). Within hours of running `make -C demos verify` over the
broader probe set in `demos/`, four typer/resolver/codegen bugs
surfaced. None of them were caught by the previous gate (two
demos + selfhost + 61-fixture suite). The pattern: extrinsic
gating (observable output) without intrinsic invariants
(typer/resolver properties).

| # | Item | SHA | Class |
|---|------|-----|-------|
| P1 | Handler-clause alias resolution gap | `10a1e7b` | typed-node threading |
| P2 | Protocol-op shadows local parameter | `257b247` | scope tracking |
| P3 | Lambda rows closed by default | `b528655` | row openness |
| P4 | Closure capture of var-bound aliases | `4c61184` | scope tracking |

All four are instances of the same systemic failure: a 22000-line
single-file compiler with no shared AST visitor; each pass
duplicates the structural recursion manually and each must
independently remember every invariant other passes already
enforce.

### Linus + Eric review verdict

Both agents converged on the diagnosis. The missing abstraction is
a **generic fold/map over the AST** ŌĆö a single `map_expr_kind(k,
f)` that recurses through every structural position. Every pass
then implements only the cases it transforms; structural cases
fall through automatically. Adding a new `ExprKind` variant would
break the compiler at one fold site, not silently degrade eight
independent walkers.

Linus surfaced four lurking suspects by inspection (each is a
`_ -> k` catch-all in a manual walker with sub-expressions
inside):

1. **`desugar_interp_decls`** ŌĆö walks `EStr` to lift `#{...}`
   fragments without scope tracking. Any interp body that shadows
   a protocol op name or references a handler alias is silently
   wrong.
2. **`pcs_rewrite_kind`** (Perceus) ŌĆö `ELitUnit` is `_ -> k`. The
   inner expression (e.g., `100.0<USD>` as the arg to
   `show(...)`) does not get the `__perceus_dup` wrapping when
   it should. Latent RC bug; inert until the emitter actually
   honors RC discipline.
3. **`rename_proto_calls_kind`** ŌĆö same `ELitUnit -> _ -> k`.
   Protocol-op references inside dimensioned literal
   sub-expressions escape the rename.
4. **`synth_lambda` doesn't clear `clause_resume`** ŌĆö a lambda
   inside `raise(resume) -> { ... }` body inherits the outer
   clause's `clause_resume`. A `resume(v)` inside the lambda
   would type-check against the wrong answer type.

### Revised closure criterion (three levels)

1. **Automated, mechanical**: `make selfhost` + `make -C stage2
   selfhost-llvm` fixed point + `make test` (incl.
   `test-demos-core`).
2. **Audit + rep invariant verifier**: every `_ -> k` wildcard in
   every AST walker pass must be justified or made explicit.
   Three rep invariants must be verifiable by walkers:
   - (a) every reconstructed `EHandle` uses typed clauses, not
     the originals;
   - (b) every `EVar(name)` rewritten by `rename_proto_calls` has
     `name` outside every enclosing binder;
   - (c) every `TyFnT` produced by `synth_lambda` for an inferred
     row has an open tail.
3. **Demo gate**: `make -C demos verify` produces no new FAIL
   relative to the milestone start. Plus: at least one
   aspirational program *not* in the milestone plan must compile,
   forcing the designer to exercise paths the milestone did not
   contemplate.

### Why only invariant (b) became a runtime walker

Eric proposed three rep invariants. Implementation surfaced that
only one is honestly verifiable post-finalise; the other two are
**construction-time** properties that a post-hoc walker either
cannot detect or over-fires on.

**(a) "Every Expr post-finalise has `Some(_)` ty annotation"** ŌĆö
**not verifiable**. Post-typer rewrite passes
(`try_rewrite_proto_call`, `try_rewrite_show_dim_real`, the AST
splicer in `desugar_interp`) construct fresh `mk_expr(...)` nodes
whose `ty` is `None` by design ŌĆö the rewriters are syntactic, not
typed. The original bug class (`10a1e7b`, discarded typed clause
body) is covered by the audit done in `7bd5a68` and by the
construction-site fix in `synth_handle`.

**(b) "Every `EVar(__proto_X)` reference has `X` outside every
enclosing binder"** ŌĆö **verifiable, implemented**. A post-finalise
walker tracks lexical scope and reports any dispatcher reference
shadowed by a local binder. Validated by temporarily reverting
the scope check in `rename_proto_calls` and observing the walker
fire on `examples/protocols/m12_8_x_invariant_b_smoke.kai`. The
walker is wired into `compile_source` after
`validate_resolved_protocols`.

**(c) "Every `synth_lambda`-produced `TyFnT` has open row tail"**
ŌĆö **not verifiable post-hoc**. Unification can legitimately bind
a fresh open row var to a closed row when the call site expects
no row (e.g., `list_sort_by`'s `cmp: (a, a) -> Int` ŌĆö closed
empty). Post-finalise we cannot distinguish "closed by default at
construction" from "closed by unification to a concrete callee
type". The check belongs at the construction site (an assertion
inside `synth_lambda`), not as a walker. The bug class
(`b528655`, closed-by-default) is covered by the construction-site
fix at line 14328 (`tail: Some(fr.id)`).

### Lessons learned from the invariant exercise

The post-hoc walker pattern works well for **rewrite invariants**
(properties about how a pass renames or restructures the AST),
where the property either holds in the entire post-pass AST or it
doesn't. (b) is one of these: post-rename, every `__proto_X` is
either a top-level reference (no shadowing) or a violation.

The pattern works **poorly** for **construction invariants**
(properties about how a node was built initially, before
unification or other passes altered it). (a) and (c) are
construction invariants. To verify them properly, the AST node
would need a "constructed by" tag, OR the assertion has to live
at the construction site itself.

The construction-site assertion approach is what kaikai's existing
audit pattern in `7bd5a68` already does ŌĆö read the construction
sites manually and confirm each one builds the right shape. Less
mechanical than walkers, but appropriate when the property cannot
be expressed as a closed AST predicate.

### Lessons learned (post-mortem of Core CLOSED ŌåÆ REOPEN)

The single-file 22000-line compiler has no shared AST visitor.
Every pass duplicates the structural recursion manually, with its
own `_ -> k` escape. The four bugs that surfaced were each one
instance of the same systemic failure ŌĆö every author must
independently remember every invariant that every other pass
already enforces (thread typed sub-expressions back, track lexical
scope, give lambdas open rows, exclude handler aliases from
captures).

The right long-term fix is a generic `map_expr_kind(k, f)` fold ŌĆö
recurses through every structural position automatically. Each
pass implements only the cases it transforms. Adding a new
`ExprKind` variant breaks the compiler at the one fold site
instead of silently degrading ten walkers. That refactor is its
own lane (~2-3 days, cross-cuts every pass). Pre-Core re-close
the audit + invariant-walker discipline below substitutes for
that refactor.
