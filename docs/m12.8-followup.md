# m12.8 follow-up: known v1 bugs and gaps

Issues found while writing a 70-line external demo
(`/tmp/kaikai-portfolio-demo/portfolio.kai`) using `--prelude
stdlib/protocols.kai`. None of these block Core; all of them are
real and should be fixed for Full or earlier.

Generated 2026-04-26 from empirical demo, post-m12.8 + m12.5 + m5.x
+ m7e13 landed.

## Bug 1 — Nested derive runtime panic

**Symptom**: `panic: __protocol_dispatch_Show_show` at runtime when
calling `show(record)` where the record has a sum-type field that
also uses `#derive(Show)`.

**Repro**:
```kai
#derive(Show)
type Inner = A | B | C

#derive(Show)
type Outer = { tag: Inner, n: Int }

fn main() : Int {
  let o = Outer { tag: A, n: 42 }
  print(show(o))   # panics
  0
}
```

**Workaround**: write `impl Show for Outer` by hand, calling
`show(o.tag)` which still works on the inner sum type alone.

**Root cause hypothesis**: the post-inference protocol-call rewrite
in `lower_protocols` resolves `show(o.tag)` only at the outermost
call site; the nested `show(...)` synthesised inside the
auto-generated `impl Show for Outer` body doesn't see the same
context because the `#derive` expander runs at parse-time and the
typer hasn't pinned the field type yet.

**Suggested fix**: have the `#derive` expander emit explicit
`__pimpl_<P>_<FieldType>_<op>` calls instead of bare `show(...)`,
or re-run the protocol-call rewrite after `#derive` expansion.

**Severity**: high. Common pattern (record-of-sum-type), confusing
panic message that doesn't point at the source.

## Bug 2 — Protocol call inside higher-order lambda panics

**Symptom**: same panic message, triggered inside a lambda passed to
`each` / `map` / `filter`.

**Repro**:
```kai
type Tx = { n: Int }
impl Show for Tx { fn show(t: Tx) : String = int_to_string(t.n) }

fn main() : Int {
  let xs : [Tx] = [Tx { n: 1 }, Tx { n: 2 }]
  each(xs, (t) => print(show(t)))   # panics on first call
  0
}
```

**Workaround**: hoist the lambda body to a named function.
```kai
fn print_tx(t: Tx) : Unit = print(show(t))
each(xs, print_tx)   # works
```

**Root cause hypothesis**: `lower_protocols` walks the top-level fn
bodies and rewrites `show(t)` to `__pimpl_Show_<T>_show(t)` based on
the typer's pin for `t`. Inside a lambda value passed to a generic
op (`each : ([a], (a) -> Unit) -> Unit`), the lambda body's typer
context may carry `t : a` (polymorphic) at the moment the rewrite
runs, so the lookup falls through to the dispatcher panic.

**Suggested fix**: run the protocol-call rewrite *after* lambda
specialisation/monomorphisation, or pass the inferred concrete `t`
type into the lambda body before the rewrite.

**Severity**: high. Idiomatic functional code (`each`/`map`/`filter`
with a lambda) is unusable with protocols.

## Bug 3 — LLVM backend prints `Real<unit>` as `0`

**Symptom**: `real_to_string` (or `show` for `Real`) returns `"0"`
for any `Real<u>` value when compiled with `--emit=llvm`. The C
backend prints the actual value.

**Repro**:
```kai
unit USD
fn main() : Int {
  let x : Real<USD> = 100.0<USD>
  print(real_to_string(unitless(x)))   # C: "100", LLVM: "0"
  0
}
```

**Root cause hypothesis**: either the LLVM lowering of
`Real<unit>` literals stores zero (Real boxing in LLVM ignores the
literal value because units' erasure happened at the typer but the
LLVM emitter still routed through a unit-aware path), or the
`unitless` cast inserts an indirection in LLVM that loses the
payload.

**Suggested fix**: audit `llvm_emit_real` and the unit-erasure path
between typer and emit. The C backend is the verified reference;
emit the same lowering.

**Severity**: high. UoM is one of the headline features for fintech
adoption; both backends must agree.

## Bug 4 — `#{x}` interpolation does not dispatch via `Show`

**Symptom**: `"...#{x}..."` always uses the runtime universal printer
(`kai_to_string`), even when `T` has `impl Show`. The user's custom
`show(x)` is silently bypassed.

**Repro**:
```kai
#derive(Show)
type Money = { amount: Int, currency: String }
impl Show for Money {
  fn show(m: Money) : String = int_to_string(m.amount) ++ " " ++ m.currency
}

fn main() : Int {
  let m = Money { amount: 100, currency: "USD" }
  print("Balance: #{m}")   # prints garbage / generic repr
  print("Balance: " ++ show(m))   # prints "Balance: 100 USD"
  0
}
```

**Workaround**: use `++ show(m)` explicitly.

**Severity**: medium. The point of `#{x}` is brevity; bypassing
`Show` defeats the protocol design. Reported as v1 limitation by
the m12.8 agent in its experience report.

**Suggested fix**: lift interp parts into the pre-inference AST so
the post-inference rewrite finds them and dispatches via `Show`
when an impl exists.

## Gap 1 — No `impl Show for Real<u: Unit>` in stdlib

Currently `unitless()` is required to print a `Real<u>` value
because there is no `Show` impl for the parametric `Real<u>` type.

**Repro**:
```kai
unit USD
fn main() : Int {
  let x : Real<USD> = 100.0<USD>
  print(show(x))   # error: no impl of Show for Real<USD>
  print(show(unitless(x)))   # works
  0
}
```

**Suggested fix**: add to `stdlib/protocols.kai`:
```kai
impl Show for Real[u: Unit] {
  fn show(x: Real[u]) : String = real_to_string(unitless(x)) ++ " " ++ unit_name(u)
}
```

This requires:
- A `unit_name[u: Unit]() : String` builtin (or compiler-provided
  reflection).
- The polymorphic-impl machinery on parametric target types — m12.8
  agent claimed it works ("tested implicitly") but no fixture exists.

**Severity**: medium. The fix unlocks `print("Balance: #{x}")`
ergonomics together with Bug 4's fix.

## Bug 5 — `show(x)` inside `impl Show for X` looks recursive

**Symptom**: ergonomic, not functional. Inside an impl body the
nested protocol calls dispatch to *other* impls, but visually they
look like a recursive call to the function being defined.

**Repro**:
```kai
impl Show for Tx {
  fn show(t: Tx) : String =
    show(t.kind) ++ "  " ++ show(t.amount) ++ " USD"
  # the two `show(...)` look like recursive calls into this same fn,
  # but they actually resolve to __pimpl_Show_TxKind_show and
  # __pimpl_Show_Real_show after the post-inference rewrite.
}
```

**Severity**: medium. Not a bug per se — the resolver does the
right thing — but the surface syntax misleads every reader who
hasn't internalised the protocol-call rewrite. Anyone debugging an
infinite-recursion problem in their own impl Show will look at this
first and waste time.

**Suggested fix**: ship a postfix method-call sugar `x.show()` that
desugars to `show(x)`. Visually unambiguous that the dispatch is on
the receiver, not the function being defined.

```kai
impl Show for Tx {
  fn show(t: Tx) : String =
    t.kind.show() ++ "  " ++ t.amount.show() ++ " USD"
}
```

This requires:
- Lexer: no change (`.` already a token).
- Parser: extend `parse_postfix_rest` to accept `.ident()` where
  `ident` resolves to a protocol op of the receiver's type.
- Typer: at the post-inference rewrite, recognise the method-call
  shape and treat it identically to `show(x)`.

The m7f follow-up §"method refs as values" mentions related
machinery; a postfix `x.op()` sugar lands on the same plumbing.

**Severity**: medium-high for ergonomics + LLM authorability —
agents reading code with `show(x)` inside `impl Show for X` will
flag false-positive recursion warnings.

## Bug 8 — `unitless` leaks typer-internal erasure as user-facing fn

**Symptom**: `unitless[u: Unit](x: Real<u>) : Real` is exposed in the
prelude as if it were an ordinary library function, but it is
actually implemented as an emit-time identity special case (no
`kai_prelude_unitless` runtime function exists). Side effects of
the leak:

1. **Cannot be used as a first-class value**: `let f = unitless`,
   `xs |> unitless`, `map(xs, unitless)` all fail at C compile time
   with `use of undeclared identifier 'kai_unitless'`.
2. **Conceptual confusion**: a user reading the prelude sees
   `unitless` as a "library helper" comparable to `int_to_string`,
   when it is actually a typer-level erasure operation.
3. **Encourages bad patterns**: users reach for `unitless(x) ++ "
   USD"` when the design-intended idiom is `print("#{x}")` once
   `Show for Real[u: Unit]` lands (Gap 1).

**Repro**:
```kai
unit USD
fn main() : Int {
  let p : Real<USD> = 100.0<USD>
  let f = unitless              # error at C compile: kai_unitless undefined
  print(real_to_string(f(p)))
  0
}
```

**Root cause**: m12.5 implementation chose to elide the runtime
function ("`unitless()` runtime representation: implemented as
identity special-case in both C and LLVM emitters — no
`kai_prelude_unitless` runtime function. Saves a runtime call and
matches the codegen-erasure invariant exactly.") The choice is
sound for codegen efficiency but the function should not have
been listed in the prelude as a user-callable value.

**Suggested fix**:

1. Remove `unitless` from the user-visible prelude (`prelude_table`
   in `stage2/compiler.kai` and the equivalent in stage 0/1 if
   referenced).
2. Keep the typer/lower pass internal logic that performs erasure
   (now triggered by an internal AST node or compiler-only
   intrinsic, not by a user-callable name).
3. After Bug 4 + Gap 1 land, the design-idiomatic way to "print a
   dimensioned value" is `print("#{x}")` and the design-idiomatic
   way to "convert between dimensioned values" is multiplication by
   a unit-bearing factor. Neither requires a user-facing `unitless`.
4. Delete `examples/units/m12_5_unitless_escape.kai` and the
   `m12_5_print_real_unit.kai` fixture's reliance on
   `unitless` (rewrite to use `Show for Real[u: Unit]`).
5. If a legitimate edge case requires breaking the unit barrier
   later (FFI boundary, manual numeric conversion), introduce it
   as an explicit escape with a name that signals the contract
   break (`discard_unit_unsafe[u: Unit](x: Real<u>) : Real` or as
   an FFI-only keyword). Do not re-add a name that sounds innocent
   like `unitless`.

**Severity**: high (correctness — first-class use crashes at C
compile time, surface inconsistency that confuses LLM and human
readers alike).

**Status**: **Fixed** in `f0acda2` (foreground, not via lane). The
fix removed `unitless` from `prelude_names` and the TyEntry table;
kept the emit special cases as dead code; deleted the two affected
fixtures (`m12_5_unitless_escape.kai`, `m12_5_print_real_unit.kai`);
added negative fixture `m12_5_unitless_unavailable.kai`. Stage 2
self-host stayed green, confirming the compiler does not use
`unitless` internally — it was pure user-space leak.

## Possible audit (not urgent, post-Core)

Other prelude entries that may be removable as redundant ergonomic
sugar (NOT leaks like Bug 8):

- `string_concat(a, b)` — `a ++ b` (m7d §23) covers this.
- `string_concat_all(xs)` — `string_join("", xs)` covers this.
- `array_make` / `array_get` / `array_set` / `array_grow` — `a[i]`
  and `a[i] := v` (m7b §6) cover the access patterns; the
  constructor (`array_make`) might still be needed.
- `int_to_string` / `real_to_string` — if `Show.show(x)` is
  ergonomic enough post-Bug-4 and Gap-1, these become obsolete at
  user-call sites (still needed inside `impl Show`).
- `string_to_int` / `string_to_real` — analogous, replaced by
  `Serialize.from_string(s)` in the Serialize protocol once
  return-type-driven dispatch lands.

These are **ergonomic tightening**, not bug fixes. Worth a
post-Core audit lane.

## Composite consequence

Together these four bugs + one gap make the demo's balance line
read:

```kai
print(show(unitless(balance(txs))) ++ " USD")
```

instead of the design-intended:

```kai
print("Balance: #{balance(txs)}")
```

Each fix is independent and small (~50-100 LOC each in stage 2).
The combined improvement is significant for user-facing ergonomics
and for the LLM-authorability bet (less "interpret the panic"
needed).

## Suggested lane: m12.8.y — protocols hardening

Bundle into one ~1-2 day lane after m12.8.x:

1. Bug 1: nested-derive dispatcher — reorder rewrite vs derive
   expansion.
2. Bug 2: lambda-body rewrite — run rewrite after lambda spec.
3. Bug 3: LLVM Real<unit> emit — backend parity audit.
4. Bug 4: `#{x}` auto-Show — interp lifting.
5. Gap 1: `impl Show for Real[u: Unit]` + `unit_name` builtin.
6. Bug 5: postfix `.show()` method-call sugar — `parse_postfix_rest`
   extension + post-inference rewrite recognition.

Each item can land independently; bundling amortises the ramp-up
cost on the protocols subsystem.

User decision (2026-04-26): this lane goes to **Core**, not Full.
Reason: a 70-line external demo surfaced all five issues immediately,
which means an external user evaluating kaikai for the first time
hits them on day 1. Core promises "protocols complete"; with these
unfixed it promises "protocols half-broken-but-with-workarounds".
